# Chat Log To Article

Agent とのやりとりや作業ログを、Zenn 記事の素材に変換する。
目的は、ログの要約ではなく、「同じように困る人に役立つ記事の核」を見つけること。

## この skill の位置づけ

軽量な 3 skill フローの入口として使う。

```text
作業ログ / agent との会話
  -> chat-log-to-article
  -> zenn-article-writer
  -> audience-review
```

この skill では記事本文を完成させない。
記事にする価値がある部分を選び、構成と材料に分ける。

## 抽出するもの

- 何に困っていたか
- なぜそれが地味につらいのか
- 何を試したか
- どこで迷ったか
- 何を採用し、何を捨てたか
- 最終的にどうなったか
- 同じ読者が真似できる手順や考え方は何か
- 筆者本人がそのテーマを書く意味はどこにあるか

## 捨てるもの

- 単なるコマンド実行ログ
- agent との細かいやりとりの往復
- 記事の本筋に関係しない失敗
- 読者が知らなくても困らない内部事情
- 長すぎる前置き
- 「便利でした」だけで終わる感想

ただし、失敗や迷いが読者の役に立つ場合は残す。

## 記事化できるかの判断

記事になりやすいログ:

- 自分が困ったことから始まっている
- 調べても一発でわからなかった
- 最終的に小さくても運用が固まった
- 同じ状況の人が真似できる
- 既存情報に対して、自分なりの整理がある

記事にしにくいログ:

- 結論だけで、困りごとが見えない
- 単なる作業代行になっている
- 読者に渡せる判断基準がない
- 個別環境に閉じすぎている

記事にしにくい場合も、足りない材料を明確にする。

## 出力形式

次の形で出す。

1. 記事の核
   - この記事で一番渡したいことを 1 文で書く
2. 想定読者
   - どんな人が読むと助かるかを書く
3. 冒頭に置く困りごと
   - 読者が「あー、それある」と思える形にする
4. 使う材料
   - 本文に入れるログ、判断、手順を箇条書きにする
5. 捨てる材料
   - 記事には入れないものを明示する
6. 推奨構成
   - 見出し案を出す
7. タイトル案
   - 3案くらい出す

## 注意

- ログの時系列に引っ張られすぎない
- 技術的に盛りすぎない
- 筆者の困りごとや温度感を消さない
- 読者が真似できる粒度まで落とす
- 必要なら「この記事にするには、まだこの情報が足りない」と言う
