# Zenn Article Writer

Zenn 記事を作るときは、技術的に正しいだけでなく、「自分が困ったことを、同じように困る人が自走できる形に整理する」記事にする。

## 参照する文体

このリポジトリでは、特に以下の流れに寄せる。

- `articles/105feac3f8e726830f8c.md`
- `articles/9e60519d9a506699d701.md`
- `articles/7c68a24a032b4d.md`

必要なら対象記事と上記の記事を読み、文体と構成の距離を確認してから編集する。

## 文体の特徴

- まず自分の困りごと、違和感、めんどくささから入る
- 「自分はこうしている」「自分の場合はこうだった」という実感を残す
- 断言しすぎず、「良さそう」「しっくりきてる」「気がしてる」くらいの温度感を使う
- ただし、手順や結論は曖昧にしすぎない
- 読者に上から教えるより、調べたことを整理して共有する
- 完璧なベストプラクティスではなく、現時点の運用として書く
- 難しい言葉より、普段の開発者同士の会話に近い言葉を使う
- 「めんどくさい」「つらい」「ありがたい」「いい感じ」などの自然な語彙を残す
- 自信がない部分は、自信がないまま書いてよい

避けること:

- マーケティングっぽい煽り
- 過剰にきれいな文章への整形
- 一般論だけで終わる構成
- 筆者の経験や判断理由を消すこと
- 「これが正解です」と言い切る記事への寄せすぎ

## 記事の基本構成

必ずこの形に固定しなくてよいが、迷ったらこの順で組む。

1. はじめに
   - 何に困ったか
   - なぜ今それを整理するのか
   - 自分の前提や経験値
2. やりたいこと
   - この記事で実現したい状態
   - 読者が読後にできるようになること
3. 背景
   - 既存情報で足りなかったこと
   - 自分が混乱したポイント
4. 本題
   - 手順、設定、考え方を小さく分ける
   - コードや設定例は必要な分だけ置く
   - なぜその形にしたかを短く添える
5. 注意点
   - ハマりどころ
   - まだ自信がないこと
   - 他の選択肢がありそうなこと
6. おわりに
   - 何が楽になったか
   - 今後変えたいこと
   - 間違いがあれば教えてほしい、くらいの余白

## 冒頭の作り方

冒頭は次の要素を入れる。

- 読者が「あー、それある」と思う困りごと
- 筆者が実際に困った場面
- この記事で整理する範囲
- 筆者の前提や自信のなさ

例の方向性:

```text
〇〇を使っていると、△△が地味につらいです。
最初は適当にやっていたんですが、しばらくすると□□が分からなくなってきました。
ということで、今はこういう形にしています。
自分の中で運用が固まってきたので整理しておきます。
```

## タイトルの作り方

タイトルは、読者の困りごとと記事の到達点がわかるものにする。

合いやすい型:

- `〇〇を完全に理解した`
- `めんどくさがりでもできる〇〇`
- `〇〇を使った△△のやり方`
- `〇〇を△△にまとめる`
- `〇〇で困ったので整理した`

強くしすぎる必要はない。少しゆるくても、読者に刺さる具体性を優先する。

## レビュー観点

下書きを作ったら、文章のうまさより先に読者への届き方を見る。

- 誰に読んでほしい記事か
- 読者にとって何がおもしろいか、何が役に立つか
- 読者にとっての当たり前水準は適切か
- 読者が持っている前提情報を正しく仮定できているか
- 読者になじみのある言葉で書かれているか
- 情報量は読者にとって適切か
- 情報の並べ方はわかりやすいか
- タイトルと冒頭で読む理由が伝わるか
- 読後に読者がどうなってほしい記事か
- なぜこの書き手がこのテーマを語る意味があるか

## 作業手順

1. 対象記事、メモ、またはユーザーの素材を読む
2. 想定読者と読後の状態を 1 文で置く
3. 筆者の困りごと、試行錯誤、判断理由を探す
4. 構成を「困りごと -> やりたいこと -> 背景 -> 手順 -> 注意点 -> おわりに」に近づける
5. 文体を整えすぎず、筆者の温度感を残す
6. 最後にレビュー観点で読み直し、必要ならタイトルと冒頭を直す

## 出力

ユーザーが編集を求めている場合は、対象ファイルを直接編集する。
レビューや構成案だけを求めている場合は、次の順で短く出す。

1. 想定読者
2. 読む理由
3. 推奨構成
4. タイトル案
5. 直すと効くポイント
