# Audience Review

対象ファイル: `$ARGUMENTS`

技術記事を、文章表現の巧拙ではなく、「想定読者にとって本当に読まれるか・役に立つか」という観点でレビューする。

## 重視すること

- 技術的な正しさの指摘は最低限にとどめる
- コードの細かい改善提案よりも、読まれにくさ / 読まれやすさの構造を指摘する
- 書き手を否定せず、改善につながる具体的なフィードバックにする
- 筆者の経験、温度感、言い切りすぎない良さを残す
- ただし、読者に届きにくい曖昧さは具体的に指摘する

## 必ず見る観点

1. 誰に読んでほしい記事か
2. 読者にとって何がおもしろいか、何が役に立つか
3. 読者にとっての当たり前水準は適切か
4. 読者が持っている前提情報を正しく仮定できているか
5. 読者になじみのある言葉で書かれているか
6. 情報量は読者にとって適切か
7. 情報の並べ方はわかりやすいか
8. タイトルと冒頭で読む理由が伝わるか
9. 読後に読者がどうなってほしい記事か
10. なぜこの書き手がこのテーマを語る意味があるか

## レビュー手順

1. 対象記事を読む
2. 想定読者を推定する
3. 冒頭、タイトル、見出しだけで読む理由が伝わるかを見る
4. 本文の情報量と順番が、想定読者の理解順に合っているかを見る
5. 筆者自身が語る意味が出ている箇所と、一般論に寄りすぎた箇所を分ける
6. 改善インパクトが大きい順に指摘する

## 出力形式

1. 記事の良い点（読まれやすい理由）
2. 読まれにくくなっている原因（構造的な問題）
3. 特に改善インパクトが大きいポイント

## 出力の注意

- できるだけ対象記事の見出しや本文の位置を示して指摘する
- 抽象論だけにせず、どう直すと読まれやすくなるかを書く
- 細かい表記ゆれや日本語の添削を主目的にしない
- 重要度の低い指摘を大量に並べない
- 改善案は、筆者の文体を消さない方向にする
