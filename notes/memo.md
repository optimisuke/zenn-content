### Zenn 記事の図解

Zenn 記事に図を入れるときの、今回の作業で得た知見。

#### 基本方針

- 図は本文の代わりに説明しきるものではなく、本文を読む前後に構造をつかむためのものとして作る。
- 図のタイトル直下に長い補足文を置くと、本文と役割が重なって読みにくくなる。
- ただし、各ボックスや矢印の説明まで削ると、図単体で意味が追えなくなる。
- タイトルは短くし、説明はボックス内・矢印ラベル・本文側に分散させる。

#### 言葉の揃え方

- 図中の言葉は、本文で使っているキーワードと揃える。
- 記事内で重要な言葉を図にも出すと、読者が本文と図を行き来しやすい。
- 事実と違う表現や、実際にはやっていない作業名を図に入れない。
- 抽象語だけでまとめず、記事固有の言葉を少し残す。

今回の例:

- `AIコメント`
- `会話`
- `ポートフォリオ`
- `一発生成`
- `対話的な修正`
- `terraform plan`
- `terraform apply`

#### レイアウト

- 長い言葉はボックスからはみ出しやすい。
- はみ出す場合は、短くする、改行する、フォントサイズを下げる、ボックスを広げる。
- ボックス内の主語と補足は、フォントサイズと位置で差をつける。
- 図は情報量を増やしすぎず、1枚につき伝える構造を1つに絞る。

#### 画像形式

- SVG は環境によって記事上で読み込まれないことがある。
- Markdown からは PNG または JPEG を参照する方が安全。
- SVG は編集用の元データとして残しておくと、あとから文言や配置を直しやすい。
- PNG 化した後は、変換後の画像を必ず目視確認する。
- Quick Look の変換は正方形サムネイルになりやすいので、記事用には headless Chrome などで SVG の縦横比どおりに変換する。

#### チェック項目

- Markdown の参照先が PNG/JPEG になっているか。
- 元 SVG と生成 PNG の両方が残っているか。
- 本文中の用語と図中の用語がズレていないか。
- 図中の文字がボックスからはみ出していないか。
- 図に入れた説明が、本文と矛盾していないか。
- 図が「説明の圧縮」ではなく「構造の理解」に効いているか。
