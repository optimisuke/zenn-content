# macOS Tahoe で micro:bit を CLI 操作する：マスストレージ書き込み問題を mpremote で解決した

macOS Tahoe で micro:bit を CLI から書き込もうとしたら、マスストレージ経由の書き込みが全滅した。調査と試行錯誤の末、**MicroPython + mpremote** によるシリアル転送で CLI ワークフローを確立できた。その記録。

**この記事で分かること：**

- macOS Tahoe で micro:bit の hex 書き込みが壊れている原因
- pyocd や pxt が使えない理由
- 初回だけブラウザ、以降は `make flash` で完結する CLI ワークフロー

## 環境

- macOS Tahoe 26.3.1 (Darwin 25.3.0)
- micro:bit v2.1.2（nRF52833）
- ring:bit2（Elecfreaks 製の micro:bit 拡張ボード。サーボポート P0〜P2 など Grove 端子を備える）
- サーボ × 3（P0 / P1 / P2）
- Python 管理: uv
- Node.js 管理: volta

## micro:bit の書き込み構造（前提知識）

後述のハマりと解決策を理解するための前提。急ぎの人はスキップして「ハマったこと」から読んでも問題ない。

### USB 接続で見える3つの口

micro:bit を USB 接続すると、1本のケーブルで3つのインターフェースが見える。

```
micro:bit (USB)
├── Mass Storage (MSC)  → MICROBIT ドライブ（.hex をコピーして書き込む）
├── USB HID (CMSIS-DAP) → デバッグ・書き込みプロトコル（WebUSB / pyocd）
└── CDC Serial (UART)   → シリアル通信（MicroPython REPL / mpremote）
```

今回の問題はマスストレージ（上段）が壊れていること。解決策は CDC Serial（下段）を使う mpremote。

### MicroPython のフラッシュ書き込みは2段階

```
フラッシュの中身
┌────────────────────────────────┐
│ MicroPython ランタイム（内蔵）  │ ← ブラウザで一度だけ焼く
│  └ microbit モジュール          │
│     ├ pin0, pin1, pin2         │
│     ├ display, button_a ...    │
├────────────────────────────────┤
│ ファイルシステム                │
│  └ main.py                    │ ← mpremote で差分更新できる
└────────────────────────────────┘
```

ランタイムさえ入れば、以降の `main.py` 更新は CDC Serial 経由で完結する。

#### 2つの Python は別物

micro:bit には Python 系の環境が2つある。

|          | MicroPython              | MakeCode Python           |
| -------- | ------------------------ | ------------------------- |
| 提供元   | micro:bit 財団           | Microsoft                 |
| 実体     | Python インタープリタ    | TypeScript の別表記       |
| API      | `from microbit import *` | `basic.showString()` など |
| REPL     | ✅ あり                  | ❌ なし                   |
| 実行速度 | 遅い（解釈実行）         | 速い（ネイティブ ARM）    |
| エディタ | python.microbit.org      | makecode.microbit.org     |

MakeCode Python はコンパイル後に TypeScript と同一の hex になる。

#### hex ファイルが大きい理由

MicroPython の hex は **1.8MB** あるが、これは Intel HEX 形式（テキスト）のため。実バイナリは約 1/3 になると考えられ、micro:bit v2 の 1MB フラッシュに収まる計算になる。また、Universal Hex 形式（v1/v2 両対応を1ファイルに収める形式）も理由の一つ。

## ハマったこと

### macOS と micro:bit のマスストレージ相性問題

この問題は macOS 13 Ventura（2022年）から報告されており、Tahoe 固有の問題ではない（[DAPLink Issue #982](https://github.com/ARMmbed/DAPLink/issues/982)）。macOS が仮想 FAT デバイスへの extended attributes の書き方を変更したことが根本原因とされており、Apple は 13.2 で一度修正したが、その後のバージョンでも同様の問題が再発している。

今回の環境（macOS Tahoe 26.3.1）でも同様の症状が発生した。ただし、**Tahoe で問題が再発した正確な原因は未確認**であり、DAPLink のファームウェアバージョンや他の要因の可能性も排除できない。

micro:bit の**仮想 FAT**（実体のないストレージを DAPLink チップがエミュレートしているもの）に対してマスストレージ経由の書き込みがことごとく失敗した。

#### 症状

```bash
# cp: 無限ハング（U ステート - kill -9 でも止まらない）
cp micropython.hex /Volumes/MICROBIT/micropython.hex

# uflash: Device not configured
Error flashing main.py to microbit: [Errno 6] Device not configured

# Finder のドラッグ＆ドロップ: "Preparing to copy..." のまま止まる
```

#### 原因の掘り下げ

```bash
mount | grep MICROBIT
# → /dev/disk4 on /Volumes/MICROBIT (msdos, local, nodev, nosuid, read-only, noowners, noatime, fskit)
```

`read-only` かつ `fskit` フラグが立っている。FAT の dirty bit が影響して read-only マウントになっている可能性が高いが、正確な原因は未確認。

以下でいったん rw マウントできるが、書き込み自体は依然ハングする：

```bash
diskutil unmount disk4
diskutil mount -mountOptions rw disk4
```

extended attributes を除いてコピーする `cp -X` も試みたが、同様にハングした（macOS 13 Ventura での報告では有効な workaround とされているが、今回の環境では効果なし）。

#### pyocd（CMSIS-DAP 経由）も無理だった

CMSIS-DAP は HID 経由でデバッグ・書き込みを行うプロトコル。`pyocd` を使えばマスストレージを使わずに書き込めると思ったが、2つの壁があった。

1. **ブラウザが USB を掴んでいると `Unable to claim interface`**
   - MakeCode や python.microbit.org が WebUSB でペアリングしていると競合する
2. **sudo で実行しても `flash erase sector failure (0x67)`**
   - micro:bit v2 の DAPLink が直接 SWD アクセスをブロックしている模様
   - DAPLink 経由（マスストレージか WebUSB）でしか書き込めない設計と思われる

```bash
sudo .venv/bin/pyocd flash -t nrf52833 /tmp/micropython_v2.hex
# → flash erase sector failure (address 0x00000000; result code 0x67)
```

#### pxt（MakeCode CLI）も同じ壁

MakeCode は TypeScript → ARM ネイティブコードにコンパイルするため hex が小さくなることを期待したが、CODAL ランタイムがバンドルされるため 1.2MB 程度にしかならなかった。書き込み方法はマスストレージ経由なので同じ問題が発生。

## 解決策

### 初回のみブラウザで MicroPython を書き込む

[python.microbit.org](https://python.microbit.org) でコードを書いて "Send to micro:bit" する。これは WebUSB（CMSIS-DAP の HID インターフェース）を使うため、DAPLink が「正規の書き込み」として受け付ける。

**ブラウザは初回のみ。以降は mpremote で CLI から完結できる。**

### mpremote でシリアル経由のファイル転送

MicroPython が入った状態であれば、CDC シリアル経由でファイルを転送・実行できる。

```bash
uv add mpremote
```

#### ハマりポイント: MICROBIT ドライブをアンマウントしないと失敗する

MICROBIT ドライブがマウントされたまま mpremote を使うと、USB HID と CDC が競合してシリアルが文字化けする。

```
b'raw REPL; CTRL-B o exit\r\n>'  # "to" が "o" になっている
```

**毎回アンマウントが必要：**

```bash
diskutil unmount disk4
```

#### ハマりポイント: `mpremote cp` がタイムアウトする

```bash
uv run mpremote cp main.py :main.py + reset
# → timeout waiting for second EOF reception
```

`cp` コマンドは MicroPython v1.18 との相性でタイムアウトすることがある。`exec` でファイル内容を直接書き込む方法で回避できる。

#### ハマりポイント: exec 内の open() は micro:bit 側で実行される

```bash
# ❌ これは micro:bit 側の古い main.py を読んでしまう
uv run mpremote exec "f=open('main.py','w'); f.write(open('main.py').read()); f.close()"
```

ホスト PC 側でファイルを読んで文字列として埋め込む必要がある：

```bash
# ✅ ホスト側で repr() に変換してから渡す
uv run mpremote exec "f=open('main.py','w'); f.write($(python3 -c "print(repr(open('main.py').read()))")); f.close()" + reset
```

## 最終的なワークフロー

```
初回のみ: python.microbit.org でブラウザ書き込み（MicroPython ランタイムを載せる）
    ↓
以降: make flash で CLI から更新
```

### Makefile

```makefile
PORT := /dev/cu.usbmodem1102

flash:
	diskutil unmount disk4 2>/dev/null || true
	uv run mpremote connect $(PORT) exec \
		"f=open('main.py','w'); f.write($(shell python3 -c "print(repr(open('main.py').read()))")); f.close()" \
		+ reset

repl:
	uv run mpremote connect $(PORT) repl
```

```bash
make flash   # main.py を書き込んでリセット
make repl    # 対話型 Python シェルに接続
```

## サーボ制御コード

ring:bit2 の P0/P1/P2 にサーボを接続。

```python
from microbit import *

PINS = [pin0, pin1, pin2]

for p in PINS:
    p.set_analog_period(20)  # 50Hz（サーボの標準 PWM 周波数）

def angle_to_analog(deg):
    deg = max(0, min(180, deg))
    return int(51 + (deg / 180) * 51)

def set_angle(pin, deg):
    pin.write_analog(angle_to_analog(deg))

def all_angle(deg):
    for p in PINS:
        set_angle(p, deg)

all_angle(90)
display.show(Image.YES)
sleep(1000)

while True:
    if button_a.is_pressed() and button_b.is_pressed():
        all_angle(90)
        display.show("N")
    elif button_a.is_pressed():
        all_angle(0)
        display.show("0")
    elif button_b.is_pressed():
        all_angle(180)
        display.show("1")
    sleep(100)
```

### PWM の計算

サーボは 50Hz（20ms 周期）の PWM で制御する。micro:bit の `write_analog` は 0〜1023 の値を取る。

```
1ms パルス（0度）  = 1/20 × 1023 ≈ 51
1.5ms パルス（90度）= 1.5/20 × 1023 ≈ 77
2ms パルス（180度）= 2/20 × 1023 ≈ 102
```
