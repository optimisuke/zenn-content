# はじめに

[テックルズ](https://techls.org/)は7/29・7/30に経済産業省こどもデーに出展し、「AIで創造力を爆発させよう！」というテーマで、子どもたちとAIを使って4コマ漫画づくりにチャレンジします。そこで使う4コマ漫画生成アプリ「どやねん.ai!」(とりあえずで決めた名前がそのまま) を作っているのですが、**これってContext Engineeringじゃね？**と思ったので、見えてきたことを整理します。

https://x.com/techtechtechls/status/2069975197105729911

![](/images/c4e91a7b2d6f30/overview.png)

# 作ったもの

作ったのは、人間とAIが交互に4コマ漫画を描くアプリです。

1. 人間が1コマ目を描く
2. AIが2コマ目を描く
3. 人間が3コマ目を描く
4. AIが4コマ目のオチを描く

テーマは「AIと人との共創」です。

AIが描いたコマには、画像だけでなく「1コマ目をどう理解したか」「なぜこの展開にしたか」というCaptionも残します。

単純な画像生成に見えますが、**2コマ目のAIには1コマ目を、4コマ目のAIにはそれまでの展開を渡す必要があります**。
さらに、画像生成用の指示と、人間が読めるCaptionも作らないといけません。

もっと簡単に作れると思ってたのですが思ったより難しいことに途中で気づきました😇

# 今回やったこと、やらなかったこと

今回採用したのは、プロンプトの分割、JSONによるメタデータの受け渡し、Contextの構造化、プロンプトのDB管理、Version管理、Captionの保存です。

逆に、RAG、Memory、Self Reflection、DSPy、Loop Engineering等は実施してません。それをContext Engineeringと呼んでいいのか気になるところではありますが、ContextをEngineeringしている感覚はあったのでContext Engineeringなんだと思います。

4コマ漫画には「この出力が正解」という明確な評価基準がありません。自動評価のツールDSPyとかが気にはなっていたのですが、「面白い」「オチがある」といった感覚を自動評価して最適化するより、**今回はHuman in the Loopで、人が出力を見ながら直すヒューリスティックな進め方が良い**と判断しました。（行き当たりばったりでやりましたってのをそれっぽい言葉で説明してみました）

# フローの設計

実際に組んでみると、1コマ目→2コマ目→3コマ目→4コマ目という単純な4ステップでは済みませんでした。

各コマの間には `/api/interpret`(前のコマのコメント生成)、`/api/plan`(企画・前の流れの要約)、`/api/generate`(画像生成)、`/api/destyle`(写真感の除去・漫画演出)という複数のプロンプトが挟まり、どんどん処理が細分化していきました。

こうなると、個々のプロンプトや入力より「どこからどこへ渡してつないでいくか」を気にする必要がでてきます。
**プロンプトがNodeで、Contextの受け渡しがEdgeです**。文章の集まりとしてではなく、有向グラフとして考えた方が整理しやすくなりました。（それっぽい専門用語を使ってケムに巻く作戦です）

# 思ったより複雑になり、専用の管理画面を作った

フローが増えると、デバッグもめんどくさくなります。昔考えたフローもすぐに忘れちゃいます。

結構複雑化してしまったので、生成画像がおかしいとき、原因がプロンプトなのか、渡したContextなのか、Captionなのか、JSONの欠落なのかがすぐには分かりません。

Langfuseのようなトレースツールも使っていましたが、それだけでは「いまどのSTEPでどのプロンプトが組み合わされているか」をひとめで掴むのが難しく、認知負荷が高いままでした。そこで、アプリの構造に合わせた専用の管理画面を作りました。（AIにお願いしただけではある）

各STEPで、編集できるプロンプト・コード管理の固定部分・実行時に決まる入力がどう組み合わさっているかを整理したのが、以下のキャプチャです。

![](/images/c4e91a7b2d6f30/prompt-management.png)

青いチップは編集できるプロンプト、グレーはコード管理の固定部分、点線は実行時に決まる入力(画像や企画結果)です。

右側のプロンプト設定パネルでは、プロンプト単体の編集・Version管理(v2、v4など)・変更履歴・過去バージョンへの復元ができ、直近の生成ログ(使用モデルや実行時刻)もそのプロンプトに紐づけて確認できます。

これでも複雑でわかりにくい気はしますが、すぐに思い出せるようにはなりました。

**結局ここまで作ることになったのは、フローが複雑になったから**だと思います。もっとシンプルにとは思いましたが、どうしてもこれくらいにはなるなーと感じてます。きっと、世の中のAIアプリも裏側は複雑なんだろうなと思います。

# おわりに

4コマ漫画生成アプリを作って**コンテキストのフローの可視化が大切**なことがわかりました。ないと、何が起きているかわからなくなります。

プロンプトやAI呼び出しが増えて管理がつらくなってきたら、プロンプト管理画面を作ったり、NodeとEdgeとして全体を描いてみると整理しやすいかもです。
