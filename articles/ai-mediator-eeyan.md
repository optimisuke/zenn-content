# はじめに

本業のかたわら、テックルズという有志のサークル活動？でエンジニアをしている伊藤です。テックルズは、福祉業界への深い関心を背景に集まったデザイナーとエンジニアのチームで、AIなどのテクノロジーを活用し、障害のある方々が自己の創造性と表現力を最大限に発揮できる環境の創出を目指しています。

https://note.com/techls

https://x.com/techtechtechls

テックルズの活動でも、子育ての場面でも、ずっと感じてきたことがあります。人が作ったものに対して、褒めたり意見を言うのって難しくないですか？「いいですね」だけだと薄いし、かといって「ここが面白い」と言おうとすると、的外れだったり気まずくなったりしないか、という気持ちが出てくる。さらに、障害のある方々のアート活動の現場だと、さらに難しくなります。

それが、今回紹介する取り組みをとおして、その難しさが少し変わりそうだなと思っています。以下で詳しく記載しますが、作品をAIにみせてコメントをもらうことで、「なんか違う」「そうそうそれ！」という会話が自然に始まることがわかりました。アーティストだけでなく、スタッフにとっても「何を言えばいいか」という心理的な負担がAIに移ることで、対話そのものに集中できるようになる。AIが仲介者になることで、今まで起きなかった対話が生まれる——テックルズの活動を通してこのような現象を確認できました。

奈良にある「Good Job! センター香芝（GJC）」は、一般財団法人たんぽぽの家が運営する、障害のある人が制作活動やものづくりに携わる施設です。絵画や張子、テキスタイルなど、プロのアーティストとしても活動しているメンバーが在籍しています。

ここ数年、GJC との協働を続けています。デジタルファブリケーションの導入から、アート作品の展示・発信まで、さまざまな角度から関わってきました。

たんぽぽの家との取り組みについては、以前 note でも紹介してもらいました。

https://note.com/ibm_mirai_nakama/n/n5d6cf20e080b

https://makezine.jp/event/makers-mfk2024/m0061/

## 最初のシステム

そういった背景から、2023年、作品画像をもとに生成AIがコメントを作る Web アプリを開発しました。出力するのは4つ。

- タイトル
- 作品の印象
- ユニークな点
- 次の制作へのヒント

最初はPython フレームワークの Gradio 、llamaのマルチモーダルモデル Llava（Large Language and Vision Assistant）で試作し、その後 Next.js で作り直して Vercel にデプロイしました。1キャラクターが一発で評価を返すシンプルな設計です。

# 現場で気づいたこと：AIが仲介者になる

## AIのコメントを見ながら会話が始まった

実証実験でいちばん印象に残っているのは、AIのコメントが「正しい」かどうかより、**AIのコメントをきっかけに会話が生まれた**ことでした。

「まあ、大体合っている」「そうとも言えるかな」という反応。意図通りに読み取られていない場合、「なんか違う」「意図通りに AI に読み取らせるにはどう工夫できるか」という話になります。AIが出した文章を足がかりに、作者が自分の作品について語り始めるんです。

論文では、このことを「**バウンダリーオブジェクトとしての生成AI**」と表現しました。バウンダリーオブジェクトとは、異なる立場・専門性を持つ人たちが共通して参照できる「橋渡し役」のことです。AI が生成したコメントが、アーティスト・スタッフ・鑑賞者それぞれにとって話題の起点になる、という意味で AI がその役割を担っている、という概念です。

![生成AIが仲介者になる構造。作者、スタッフ、鑑賞者がAIコメントを共通の参照点として会話する](/images/ai-mediator-eeyan/ai-as-mediator.png)

AI が間に入ることで、作者・スタッフ・鑑賞者の間に新しい対話が生まれます。AI を褒め役にすることで、人間が「この AI はどう思う？」と話しかけやすくなります。

## 職員側にもメリットがあった

AIは「アーティストを褒める機械」というより、**スタッフとアーティストの間の会話を引き出す道具**として機能していました。「AIはこう言ってるけど、あなたはどう思う？」という問いかけができる。AIが話題の起点になることで、スタッフ自身が言葉に詰まらなくて済む。

展示・発信まわりの業務にも効果があります。作品のタイトルや説明文を毎回考えるのは地味に大変な作業ですが、AIが下案を出してくれることでその負担が減る。今回のアップデートで作品情報がポートフォリオとして蓄積されるようにしたのも、こうした業務サポートを意識してのことです。

## 一発生成では足りなかったこと

ただ、課題も見えてきました。

一番多かったのが「なんか違う」という反応です。「別に悲しんでいない」「まとまっているわけじゃない」という、納得しかねる状態になることがある。「もう一度変えられる？」「別の角度で見てほしい」という声が出てきました。でも当時のシステムは一発生成のみ。修正する手段がありませんでした。

もうひとつ、**オーサーシップの問題**もありました。AIが付けたタイトルをそのまま使うと、作者本人の表現の良さが失われる懸念がある。あくまで作者本人にオーサーシップがある状態で、AIを補助的に使う設計が必要だ、という気づきです。

このような背景から、アプリを作り直すことにしました。以下、アプリのリビルドに伴う知見の共有です。

# AIコーディング縛りで作ってみた

実装は Claude Code と Codex で進めました。かなりヘビーに使い倒したので、効いた運用と効かなかった運用をまとめます。

**このセクションで触れること**

- AGENTS.md / docs/memo.md でエージェントに前提を引き継ぐ運用
- 「任せる範囲」と「自分で判断する範囲」の分け方
- worktree 並列作業とセキュリティレビューを早めに頼む話
- デモ動画・サムネなどコード以外の成果物もエージェントに任せた話

## 1. エージェントに前提を渡す

AI コーディングで一番効いたのは、エージェントに毎回説明し直さなくてよい状態を作ることでした。新しいフレームワークや破壊的変更が多いライブラリを使う場合、エージェントの学習済み知識が古いことがあります。そこを AGENTS.md や docs/memo.md で補うようにしました。

### Next.js 16: `middleware.ts` が deprecated → `proxy.ts` へ

Next.js 16 では `middleware.ts` が deprecated になり、`proxy.ts` へのリネームが推奨されています。重要な違いは動作ランタイム。Proxy（旧 Middleware）は **Node.js ランタイム**で動くため、`crypto` や `jsonwebtoken` などの Node.js モジュールが普通に使えます。認証まわりを実装するときに気づいたので、参考まで。

### `create-next-app` が AGENTS.md に AI 向け注記を自動生成する

Next.js 16 から `create-next-app` が `AGENTS.md` に以下のブロックを自動で書き込みます。

```
<!-- BEGIN:nextjs-agent-rules -->
# This is NOT the Next.js you know
This version has breaking changes — APIs, conventions, and file structure
may all differ from your training data. Read the relevant guide in
node_modules/next/dist/docs/ before writing any code.
<!-- END:nextjs-agent-rules -->
```

「お前が知っている Next.js とは違う、コード書く前にドキュメント読め」という AI エージェント向けの警告です。プロジェクト独自の `AGENTS.md` に移行するときはこのブロックを手動でマージしておかないと消えます。

### AGENTS.md にコンテキストを集約する

Claude Code とCodex は `AGENTS.md` を読みます。今回は `CLAUDE.md` から `@AGENTS.md` を参照する形にして、1ファイルに集約しました。エージェントを切り替えても文脈がそろうので、「さっきと違うことをやり始める」が減りました。

AGENTS.md には以下を書いています。

- 技術選定の方針（Vercel AI SDK に一本化、LangChain は使わない）
- `USE_MOCK=true` の使い方（LLM を呼ばずにローカルで動かす）
- テスト方針（`vi.mock` で外部依存を遮断、実装と同時にテストを書く）
- コミット前の必須チェック（`npm run test` を通す）
- Skills の使いどころ（セキュリティレビュー・PR レビューなど）

### コミットごとにメモを残す（docs/memo.md）

今回やってよかったのが、コミットのたびに `docs/memo.md` に作業メモを残す運用です。「やったこと」「詰まったこと」「次やるなら」の3点を書いてもらうだけですが、これがセッションをまたいだコンテキスト引き継ぎにすごく効いています。

AGENTS.md に「最初に `docs/memo.md` を読め」と書いておくと、新しいセッションを始めたエージェントが前回の作業状況を把握した状態でスタートしてくれます。「先週詰まったことをまた聞かれる」「同じエラーを二回調べる」がかなり減りました。

コミット前に実行する `pre-commit.sh` の中でメモの更新を促す確認を入れています。Claude Code にはフック機能もあって、もっと使いこなしたいと思いつつ、まだうまく活用できていません。

### AGENTS.md をタスクごとに育てる

今回はタスクの区切りでセッションを切り替えながら進めていました。1セッション = 1フェーズくらいのサイズ感です。セッションの終わりに「今回の詰まりどころ・気づきを AGENTS.md に追記してください」と頼むのが習慣になっていきました。

たとえば「`create-next-app` が既存ディレクトリに生成できない問題」「Mac の ARM image は Container Apps で動かない」「DATABASE_URL にクォートが入る問題」といった詰まりどころは、最初は memo.md にだけ書かれていましたが、「次のセッションで同じミスをしないために AGENTS.md にも書いておく価値があるか？」と判断しながら取り込んでいきました。

AGENTS.md が育つにつれて、エージェントが同じ方向の間違いをしにくくなっていく感覚があります。最初は薄かったドキュメントが、プロジェクトが進むほど「このプロジェクトでの正しい動き方」を伝えられるようになる。ちょっとした知識ベースというか、チームのオンボーディングドキュメントに近い感覚になってきました。

## 2. 任せる範囲を決める

全部を AI にお願いする縛りとはいえ、判断まで丸投げするとうまくいきませんでした。どこを任せて、どこを自分で決めるかを分けた方が進めやすかったです。

### エージェントに任せた範囲・自分で判断した範囲

| エージェントに任せた   | 自分で判断した                   |
| ---------------------- | -------------------------------- |
| 実装                   | SDK・インフラの選定              |
| テストコードの作成     | プロンプト設計・キャラクター定義 |
| Terraform コードの整形 | セキュリティ方針                 |
| ドキュメントの更新     | 詰まったときの根本原因の特定     |

![人間とコーディングエージェントの分担。人間が方針決定や原因特定を担当し、エージェントが実装やテストを担当する](/images/ai-mediator-eeyan/human-agent-collaboration.png)

「どの技術を選ぶか」「AIにどういう役割を与えるか」は自分で決めて、「その実装を書く」「テストを書く」はエージェントに投げる、という分担がしっくりきています。

根本原因の特定が必要な場面は、エージェントに任せるより自分で手を動かす方が早かったです。たとえば `turbopack.root` の設定が原因で開発サーバー起動時に Mac ごとシャットダウンする問題が起きました。最初は「Mac が古いのかな」くらいにしか思っていなかったのですが、自分で `npm run dev` を実行したらみるみるしようメモリが増えて落ちることを確認したことで「これだ」とわかり、エージェントに修正してもらうことができました。「何かがおかしい」という段階は自分で踏み込む、原因がわかってから任せる、という流れが安定しています。

### Azure 初体験について

今回 Azure は初めてでした。他のクラウドは使ったことがある、という状態でスタートしています。

感想としては、「なんとかなった」です。Container Apps・Blob Storage・PostgreSQL Flexible Server・Application Insights のような概念は他クラウドと対応するものがあるので、「同じようなのがあるだろう」という感覚で探しながら進めました。詰まるポイントは Azure 固有のもの（クォータ問題・プロバイダー登録・Managed Identity のパターン）が多かったですが、ドキュメントを別で読み込んだりはしていません。エラーメッセージを見ながらエージェントがいい感じに調べて解決してくれました。CLI・Terraform ともにエージェントが書いてくれるので、「自分が書く量」よりも「確認と判断」に集中できたのは楽でした。

## 3. 並列作業とレビューを回す

エージェントを複数使うと作業は進みますが、差分の監査やセキュリティ観点のレビューは必要でした。ここは「速くする」だけでなく、「壊さない」ための運用です。

### worktree で並列作業

Claude Code の worktree 機能を使うと、メインブランチを汚さずにエージェントが別ブランチで作業できます。インフラと UI を並列で進めるときに使いました。

ただし worktree 内でエージェントにコミットさせると git 権限まわりでうまくいかないことがあります。「コードを書いて `npm run build` まで確認するところで止める、コミットは自分でやる」という運用が安定していました。

また、並列で走らせると、意図せずメイン repo に直接書いていたり、古いベースから大量の差分を出していたりすることがあります。worktree エージェントの結果は、できるだけ差分を理解してから取り込むようにしました。ファイルが重複していたり、差分が妙に大きい場合は、そのままマージせず手動で必要な部分だけ拾う方が安全でした。

### セキュリティレビューも早めに頼む

今回かなり効果があったのが、実装が動き始めた段階でセキュリティレビューをエージェントに頼んだことです。アプリと Terraform を横断して見てもらい、画像アップロード、認可、Webhook、レートリミット、HTTP ヘッダー、プロンプトインジェクションなどを確認しました。Webhook は Stripe など外部サービスからアプリに届く通知、レートリミットは短時間に大量アクセスされたときに制限する仕組みです。プロンプトインジェクションは、ユーザー入力で AI の指示を上書きしようとする攻撃です。

指摘されたものの中には、言われたら確かに危ないと分かるけれど、機能実装に集中していると見落としやすいものがありました。たとえば、画像アップロードでクライアント申告の MIME type だけを信じていたこと、SVG を同一オリジンで返すと XSS の入口になり得ること、非公開作品の画像や OGP に認可漏れが起き得ること、Stripe Webhook の本番署名検証を fail closed にすべきことなどです。MIME type はファイル種別の申告、同一オリジンは同じドメインとして扱われる場所、XSS は Cross-Site Scripting の略で、悪意あるスクリプトをページ上で実行させる攻撃です。OGP は SNS でリンクを貼ったときに出るタイトルや画像などのプレビュー情報です。fail closed は、設定が足りないときに動かし続けるのではなく、安全側に倒して止めることです。

最終的には、magic bytes による画像検証、SVG の除外、非公開作品画像の所有者チェック、Webhook 署名の本番必須化、管理画面の Basic 認証、Cloudflare WAF / Rate Limit、CSP / HSTS などを入れました。magic bytes はファイル先頭のバイト列を見て本当の種類を確認する方法です。Basic 認証はブラウザ標準の簡易な ID / パスワード保護、HSTS は HTTPS 接続を強制するためのブラウザ向け設定です。完璧なセキュリティ診断ではありませんが、少なくとも「自分の目だけで見たら通してしまいそうな穴」を早めに潰せたのは大きかったです。

特にハッカソンのように短期間で公開まで持っていく場合、セキュリティは最後にまとめてやるとつらいです。ある程度動いた段階で「攻撃者目線で見て」「セキュリティ観点でレビューして」と頼むだけでも、アプリの品質がかなり変わる感覚がありました。

## 4. コード以外の成果物も任せる

今回の縛りでは、アプリ本体だけでなく、提出物まわりもできるだけエージェントに任せました。動画、サムネイルのような周辺作業も、同じリポジトリに置くと文脈を持ったまま手伝ってもらえます。

### デモ動画と YouTube サムネもエージェントで作った

提出用のデモ動画も、かなりコーディングエージェントに寄せて作りました。PowerPoint は作らず、LP をプレゼン資料として使い、Playwright でスクロールやアプリ操作を自動化し、macOS の `screencapture` で録画する形です。

ログイン済み Chrome を Playwright から操作するために CDP 接続を使いました。CDP は Chrome DevTools Protocol の略で、Chrome を外部ツールから操作するための仕組みです。ここでは WebSocket はつながるのに DevTools の HTTP エンドポイントが 404 になる、という少し分かりにくい罠がありました。なぜか再起動でなおったのでよくわからずですが。

YouTube のサムネイルは画像生成 API ではなく、HTML/CSS + Playwright スクリーンショットで作りました。1280x720 の HTML に、実際のアプリ画面のキャプチャ、吹き出し、大見出し、キャラクター画像を配置して、Playwright で PNG 化する流れです。Canva 等を開かずに、エージェントと一緒に「この吹き出しがキャラと被るから上に動かす」みたいな調整までできたのは、思ったより便利でした。

やってみて分かったのは、コーディングエージェントは「コードを書く」だけでなく、提出物まわりの作業にもかなり使えるということです。LP、録画スクリプト、字幕、YouTube サムネまで同じリポジトリに置くと、エージェントが文脈を持ったまま手伝ってくれます。

# ええやん.ai! というアプリ

## 1. 作り直したきっかけ

Vercel でデプロイして使っていましたが、使い続けるうちに「もっとこういうことをやりたい」というアイデアが増えてきました。pub/sub でイベント処理を分けたい、コンテナイメージをちゃんとレジストリで管理したい、将来的にバックエンドをもっと独立させたい、など。Vercel でもできることはありますが、クラウドプラットフォームの方が選択肢が広いと感じていました。

ちょうどそのタイミングで Azure を使ったハッカソンを見つけました。「このタイミングで移行してしまおう」と決めて、アップデートと移行をまとめてやることにしました。締め切りがあるのも着手のきっかけになりました。「いつかリビルドしよう」では動けないので。

なお今回はフロントエンドも含めて Next.js ごとコンテナに入れて Container Apps で動かす構成にしています。シンプルで動かしやすいですが、そのうちフロントと API をきちんと分けてもっとしっかりした構成にしたいとは思っています。また、今回はチャットで対話的に修正できるところまで実装しましたが、今後はもっとエージェンティックな動き——たとえばアーティストの過去作品を参照して自律的に提案したり、複数のキャラクターが連携して評価を深掘りしたり——もやってみたいと思っています。

## 2. 体験として変えたこと

### 一発生成から、対話的に修正できる形へ

チャット形式で、AIのコメントに対して「もっとポジティブに」「色使いについてもっと具体的に」と話しかけながら修正できるようにしました。「なんか違う」と思ったとき、一方的に終わらせない設計です。

今の実装では、AIはチャットで応答しながらタイトルや作品説明をツールで書き換える、というところまでです。完全に自律して動くわけではないし、ツールもまだ最小限。ただ、「AI と対話しながら作品の言葉を作る」という土台ができたことで、拡張のアイデアが広がっています。

![旧システムと新システムの違い。一発生成で終わる旧システムから、チャットで修正して保存・蓄積する新システムへ変わった](/images/ai-mediator-eeyan/old-vs-new-flow.png)

| 旧システム                  | 新システム（ええやん.ai!）           |
| --------------------------- | ------------------------------------ |
| 一発生成のみ                | チャットで対話的に修正               |
| Gradio → Vercel + LangChain | Azure Container Apps + Vercel AI SDK |
| ポートフォリオなし          | マイページ・ギャラリー機能           |

### エージェントらしさを目的にしない

ハッカソンなので「Agentic AI としてどう見えるか」は意識していました。ただ、途中で検討した案の中には、実装すると派手だけどユーザー価値に直結しにくいものもありました。

たとえば、`analyzeImage → draftEvaluation → selfCritique → refine → save` のようなマルチステップ化や、複数キャラクターが議論してモデレーターがまとめるマルチエージェント案も検討しました。デモ映えはしますが、レイテンシとコストが増えるわりに、ワークショップで使う人にとって本当にうれしいかは微妙でした。

最終的には、チャットでユーザーの意図を解釈して評価内容を書き換える tool use と、過去作品から成長を見られる memory 的な使い方を組み合わせる方針にしました。tool use は、AI が必要に応じて「タイトルを更新する」「説明文を書き換える」といったアプリ内の機能を呼び出すことです。memory は、過去の作品や評価内容を次の判断材料として使うことです。エージェントらしく見せるための処理ではなく、「もう一度変えられる？」という現場の声に応える機能から考える。この順番を外さないことがユーザーにとって重要かと思います。

### プロンプトは「評価」ではなく「会話の下書き」として考える

プロンプトまわりも、実装しながら少しずつ整理しました。プロンプトは 作品を評価するAIに渡す指示文のことです。ここを雑にすると、AI は作品を一方的に採点するような文章を出しがちです。でもこのアプリでほしかったのは、正解の評価ではなく、作者や支援者が話し始めるための下書きでした。

まず、キャラクター定義は `lib/characters.ts` に外出ししました。名前、役割、口調、評価の観点を1か所に集めておくと、評価生成とチャットの両方で同じキャラクター設定を使えます。画面上では「深読み先生」「お笑い芸人」「熱血コーチ」のように見えますが、裏側では「このキャラクターはどういう言い方をするか」を共通の定義として持っています。

評価生成では、アーティストの自己紹介文が登録されている場合はプロンプトに含めるようにしました。作品画像だけを見るより、「その人が普段どういう制作をしているか」を少しでも踏まえた方が、コメントが会話に近づくからです。さらに、過去評価の履歴も差し込むようにしました。過去の評価を古い順に渡し、「前回から何が変わったか」を内部的に `improvements` として持たせています。これは画面にそのまま出すためではなく、次の評価で「色使いが豊かになってきた」「構図の複雑さが増えている」のような文脈を出すためのメモです。

チャット側では、現在の評価内容をプロンプトに入れるようにしました。最初は「もっと詳しく」と言われても、AI がいま表示されているタイトルや説明を踏まえずに返してしまうことがありました。そこで、現在のタイトル、印象、アドバイスを明示的に渡し、「まずキャラクターらしく返事をしてから、必要な項目を書き換える」という順番を指示しました。

ここで使っているのが Tool Calling です。Tool Calling は、AI が文章を返すだけでなく、アプリ側に用意した処理を呼び出す仕組みです。今回なら「タイトルを更新する」「作品の印象を書き換える」「次の制作へのアドバイスを更新する」といった処理です。チャットで相談した結果が、吹き出しの返答だけで終わらず、UI 上の評価フィールドにも反映されるようにしました。

評価生成の出力は、zod schema を使った構造化出力にしています。zod は TypeScript でデータの形を定義・検証するライブラリです。AI に自由な文章だけを返させると、JSON パースエラーや項目抜けが起きやすくなります。そこで、`title`、`feature`、`advice`、`improvements` のような必要項目を schema として定義し、その形で返すようにしました。Azure OpenAI の structured output では optional の扱いで実際にエラーも出たので、必要な項目を schema 上も明確にするのは大事でした。

ローカル開発では `USE_MOCK=true` の分岐も用意しました。毎回 LLM を呼ぶと遅いし、コストもかかります。UI や API のつなぎ込みを確認したいだけのときは、モック応答で十分です。LLM を呼ぶ部分と、アプリとして保存・表示・更新する部分を分けておくと、開発中の確認がかなり楽になります。

## LangChain から Vercel AI SDK に一本化

最初は LangChain を使っていましたが、今回のリビルドで Vercel AI SDK に切り替えました。理由は3つ。

- 依存が減ってシンプルになる
- Azure OpenAI との相性がいい（`@ai-sdk/azure` で直接使える）
- `streamText` によるストリーミングレスポンスがシンプルに書ける

LangGraph や Azure AI Agent Service も候補に上がりましたが、前者は複雑すぎで、後者は Next.js との統合が重かったです。Vercel AI SDK が一番素直でした。

## Vercel から Azure Container Apps へ

Vercel から Azure Container Apps に移行しました。コンテナで動かすことで、インフラまわりの自由度が上がります。今回は Prisma 7 + PostgreSQL アダプターパターンを使っており、普通の Node.js プロセスとして動く Container Apps がシンプルでした。

## Cloudflare でドメイン取得と WAF をまとめる

公開用のドメインは Cloudflare で取得しました。`techls.org` を取り、サービスごとにサブドメインを切る方針にして、ええやん.ai! は `eeyan-ai.techls.org` で公開しています。

最初は DNS、つまりドメイン名をどのサーバーへ向けるかの設定が中心でしたが、公開に近づくにつれて WAF や Rate Limit も Cloudflare 側で見るようにしました。WAF は Web Application Firewall の略で、怪しいアクセスをアプリの手前で止める仕組みです。Rate Limit は短時間に大量アクセスされたときに制限する仕組みです。`/admin` はチャレンジを出す、使わない HTTP method はブロックする、`.env` や `.git` を探しに来るアクセスは落とす、といった設定を Terraform に入れました。

ただし Cloudflare の設定は、Cloudflare 上で管理している `techls.org` 全体に効きます。今後アプリが増えるなら、アプリごとのリポジトリに WAF 設定を分散させるより、Cloudflare 用の infra リポジトリに集約した方がよさそうです。今回はシンプルさを優先してモノレポで実装してました。

## Stripe で課金の仕組みを入れる

ハッカソン時点ではベータ版ですが、継続運用を考えると評価やチャットの回数に応じた課金が必要になります。そこで Stripe を使って、500円で評価クレジットを購入できる仕組みを入れました。

実装としては、Stripe Checkout で決済ページを作り、決済完了後に Stripe から届く Webhook を受け取ってユーザーのクレジットを増やします。Webhook は外部サービスからアプリに届く通知です。ここは二重付与が起きると困るので、決済セッション ID を保存して、同じ通知がもう一度来てもクレジットを増やさないようにしました。

AIがなぜかハマったのは JPY の `unit_amount` です。Stripe では日本円は小数なしの通貨なので、`unit_amount=500` が 500円です。最初にAIがうっかり `50000` にしてしまい、5万円の商品を作ってすぐ直しました。

## 本番に近づけると、アプリ外の設定が急に効いてくる

ローカルで動いているうちはアプリコードだけ見ていればよかったのですが、公開に近づくと、Cloudflare、Azure Container Apps、GitHub Actions、Stripe、Google OAuth など、アプリ外の設定が一気に効いてきます。

たとえば、Better Auth のクライアント設定で `localhost:3000` をフォールバックにしていたため、本番ログイン後の通信先がローカルを向く問題がありました。`NEXT_PUBLIC_*` は Docker build 時にクライアント JS に埋め込まれるので、本番の実行時環境変数を直しても既存 bundle は変わりません。結局、同一オリジン、つまり同じドメイン内で動かせるところは明示 `baseURL` を消す方が素直でした。

Cloudflare も同じで、Web Analytics / Browser Insights が自動注入するスクリプトが CSP に弾かれました。CSP は Content Security Policy の略で、「このページで読み込んでよいスクリプトや画像の場所」をブラウザに指示するセキュリティ設定です。Terraform で管理しているリソースだけ見ていると気づきにくいですが、Dashboard 側で有効になっている機能も本番挙動に影響します。

このあたりは、「コードは正しいのに本番だけおかしい」という形で出ます。アプリ、ビルド時 env、実行時 env、CDN / WAF、外部 SaaS の境界を分けて見る必要がある、というのは今回かなり学びでした。ここで、CDN は画像やページをユーザーの近くから配信する仕組み、WAF は Web Application Firewall の略で、怪しいアクセスをアプリの手前で止める仕組みです。

## Prisma 7 と Prisma Client の4つの罠

Prisma 7 はかなり破壊的変更があります。

**1. スキーマに `url = env(...)` が書けない**

```prisma
// ❌ Prisma 7 ではエラー
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

`The datasource property url is no longer supported in schema files` というエラーになります。代わりにアダプターパターンに移行しました。

```typescript
// ✅ Prisma 7 のやり方
import { PrismaPg } from "@prisma/adapter-pg";
import { PrismaClient } from "./generated/prisma/client";

const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL! });
const prisma = new PrismaClient({ adapter });
```

**2. `prisma.config.ts` で dotenv が自動ロードされない**

`prisma.config.ts` を使う場合、`.env` の自動ロードがなくなっています。ファイルの先頭に `import "dotenv/config"` を追加する必要があります。

**3. 生成ファイルのインポートパス**

`output = "../lib/generated/prisma"` にした場合、エントリポイントは `index.ts` ではなく `client.ts` です。

```typescript
// ✅ 正しいインポート
import { PrismaClient } from "./generated/prisma/client";
```

**4. Prisma Client の singleton は本番でも効かせる**

本番で評価 API が 500 になり、Log Analytics に `TooManyConnections` が大量に出たことがありました。原因は Prisma Client の singleton 化が本番では効いていなかったことです。singleton は「同じものを何度も作らず、1つだけ使い回す」ためのパターンです。

Next.js のホットリロード対策として「開発環境だけ `globalThis` にキャッシュする」書き方がありますが、今回は Prisma Client を Proxy 越しに扱っていたため、プロパティアクセスのたびに `getPrismaClient()` が走っていました。Proxy は、オブジェクトへのアクセスを一度横取りして処理できる JavaScript の仕組みです。本番ではキャッシュしない実装だったので、アクセスごとに `new PrismaClient()` され、新しい pg Pool が増えて PostgreSQL の接続数上限をすぐ超えました。

最終的には環境に関係なく常に `globalThis` にキャッシュする形にしました。よく見るスニペットでも、周辺の実装パターンと組み合わせると普通に事故る、という例でした。

## Docker・デプロイの詰まりポイント

**Mac の ARM image は Container Apps で動かない**

Apple Silicon Mac で `docker build` すると arm64 の image が作られます。Container Apps は `linux/amd64` が必要なので、そのまま push するとデプロイ後にエラーになります。

```bash
# ✅ amd64 向けにビルドする
docker buildx build --platform linux/amd64 -t myimage .
```

**DATABASE_URL のクォート問題**

`.env.local` の `DATABASE_URL="postgresql://..."` を shell script で取り出すと引用符が含まれる場合があります。Prisma が `"postgresql://..."`（先頭の `"` がリテラル）を接続文字列として解釈して接続失敗します。

```bash
# ❌ クォートが残る
DATABASE_URL=$(cat .env.local | grep DATABASE_URL | cut -d= -f2-)

# ✅ クォートを除去する
DATABASE_URL=$(cat .env.local | grep DATABASE_URL | cut -d= -f2- | tr -d '"')
```

**CI では Dockerfile に `prisma generate` が必要**

`lib/generated/prisma/` は `.gitignore` 対象のためリポジトリに含まれません。CI の docker build で `npm run build` の前に `npx prisma generate` を実行しないと、生成ファイルが存在せずビルドが失敗します。

```dockerfile
RUN npx prisma generate
RUN npm run build
```

**vendor-docs に公式ドキュメントを submodule で置く**

Prisma 7 は破壊的変更が多く、ネット上の情報や LLM の学習データが古いことがあります。そこで Prisma の公式ドキュメントリポジトリを `vendor-docs/prisma/` に git submodule として追加して、エージェントがコードを書く前に必ず参照するよう AGENTS.md に書きました。Vercel AI SDK も同様に `vendor-docs/vercel-ai/` に置いています。

```bash
# 初回セットアップ
git submodule update --init

# ドキュメントを最新に更新
git submodule update --remote vendor-docs/prisma
```

ただし submodule のサイズに注意が必要です。Prisma のリポジトリはフルで clone すると 1.8GB あります。

**`tsconfig.json` の exclude を忘れるとビルドエラー**

`tsconfig.json` が `**/*.ts` を include していると、`vendor-docs/prisma/` 配下の型エラーをビルド時に拾ってしまいます。`"exclude": ["vendor-docs"]` を追加して除外しておきましょう。

**.dockerignore を忘れると vendor-docs が丸ごと image に入る**

`.dockerignore` に追加しないと 1.8GB の submodule が image に含まれて ACR へのアップロードが失敗します。また Turbopack のファイルウォッチャーがこのディレクトリを監視対象にしてしまうと、開発サーバー起動時にメモリを食い尽くすこともあります（実際に Mac がシャットダウンしました）。`next.config.ts` から `turbopack.root` を削除して解決しました。

```
# .dockerignore
vendor-docs
.git
node_modules
```

## Terraform / GitOps 化

インフラは Terraform でコード管理し、GitHub Actions で CI/CD を組みました。

![Terraform と GitHub Actions のデプロイフロー。Pull Request では terraform plan、main へのマージ後に terraform apply、Docker build、Container Apps へのデプロイを行う](/images/ai-mediator-eeyan/gitops-flow.png)

**azurerm v4 の変更**

`azurerm` プロバイダー v4 では API が変わっていて、そのままでは Terraform コードが通りません。主な変更点：

- `skip_provider_registration` が廃止 → `resource_provider_registrations = "none"`
- `azurerm_cognitive_deployment` の `scale` ブロックが廃止 → `sku` ブロックに変更

**既存リソースの import**

`az` コマンドで先に作ったリソースを Terraform に取り込む場合、import ブロックの ID は大文字小文字が区別されます。

```hcl
# ✅ Microsoft.Insights（大文字小文字が正確に一致する必要がある）
import {
  to = azurerm_application_insights.main
  id = "/subscriptions/.../resourceGroups/.../providers/Microsoft.Insights/components/..."
}
```

**Managed Identity は User-Assigned で作らないと plan が通らない**

Container Apps に Managed Identity を付けて Blob Storage へのアクセス権を与えようとしたとき、System-Assigned Identity（`azurerm_container_app` のリソース内に書く形）では `terraform plan` が通りませんでした。`azurerm_role_assignment` の `principal_id` に `azurerm_container_app.main.identity[0].principal_id` を参照させようとすると、plan 時点でその Identity がまだ存在せず "Missing required argument" エラーになります。

解決策は User-Assigned Identity（`azurerm_user_assigned_identity`）を別リソースとして先に作ること。`principal_id` が plan 時から computed 属性として参照できるようになります。

**Terraform state を Azure Blob に置く**

GitHub Actions から state を参照するため、Azure Blob に remote backend を構成しています。ローカルで `terraform apply` すると state が競合するリスクがあるので、原則 PR 経由で apply する運用にしました。

## Application Insights の自動計装

`@azure/monitor-opentelemetry` を使うと Azure Application Insights に自動計装できます。最初は `node --require @azure/monitor-opentelemetry server.js` という形で試しましたが、Next.js standalone build では MODULE_NOT_FOUND になりました。

Next.js 15 以降は `instrumentation.ts` を使うのが正しいやり方です。

```typescript
// instrumentation.ts（プロジェクトルート）
export async function register() {
  const { useAzureMonitor } = await import("@azure/monitor-opentelemetry");
  useAzureMonitor();
}
```

また、`@prisma/adapter-pg` と `pg` は Node.js ネイティブモジュールなので `next.config.ts` の `serverExternalPackages` に追加が必要です。これをしないと standalone build に含まれません。

# おわりに

「AI が仲介者になる」というのは、今後いろんな場面で使えるアイデアだと感じています。

人が作ったものを褒めたりアドバイスするのは、AIではなく人間がすべきことではあるとおもいますが、難しさもあります。関係性がある分、言葉を選びすぎてしまうのです。でも AI が間に入ると、「AI はそう言っているけど、あなたはどう思いますか？」という会話ができる。評価の矢印が AI に向かうことで、人が楽になります。

障害のある人の表現活動だけでなく、子どもの絵を先生が評価するとき、部下の成果物にフィードバックするとき、似たような構造の場面はあちこちにあると思います。

施設の職員にとっても、AIが間に入ることで「何を言えばいいか」という負荷が下がり、アーティストとの関係をもっと丁寧に育てることができる。業務の効率化というより、関係の質の変化、と言った方が近いかもしれません。それが、このアプリが目指す人によりそった「AIエージェントによる業務改革」です。

今回の進化のきっかけになったのも、現場からの「もう一度変えられる？」という声でした。次のフィードバックで、またアプリは変わるはずです。施設の方から「ワークショップでも使いたい」という話も来ているので、引き続き現場と一緒に作っていきます。
