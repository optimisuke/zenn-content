# はじめに

Claude Code、Codex、IBM Bob みたいなコーディングエージェントを並べて試していると、「同じ設定を何回書くんだ」問題が出てきます。

プロジェクトのルール、コミット前に見ること、テストの実行方法、よく使うスキル。最初はそれぞれのツールにコピーすればいいんですが、しばらくすると **どのファイルが最新なのか分からなくなる** んですよね。自分も Claude Code、Codex、IBM Bob を行き来しているうちに、だんだん管理がしんどくなってきました。

ということで、今は **AGENTS.md と `.agents/skills/` に集約する** 形にしています。設定ファイルを1箇所に寄せて、各ツールからそこを読むだけです。

N番煎じな話かもしれませんが、自分の中で運用が固まってきたので整理しておきます。

# やりたいこと

やりたいことはシンプルです。

- プロジェクト共通の指示は `AGENTS.md` にまとめる
- Claude Code だけ `CLAUDE.md` から `AGENTS.md` を読み込む
- スキルは `.agents/skills/` にまとめる
- 各ツールのスキルディレクトリは symlink でそこを見る

最終的にはこういう構成にします。

```text
.
├── AGENTS.md
├── CLAUDE.md
├── .agents/
│   └── skills/
│       ├── review-pr/
│       │   └── SKILL.md
│       └── release-note/
│           └── SKILL.md
├── .claude/
│   └── skills -> ../.agents/skills
├── .codex/
│   └── skills -> ../.agents/skills
└── .bob/
    └── skills -> ../.agents/skills
```

これだけで、新しいスキルを `.agents/skills/` に置けば Claude Code、Codex、IBM Bob から同じものが見えるようになります。

# 背景

各ツールは、それぞれプロジェクトのコンテキストファイルやスキルディレクトリを持っています。

| ツール      | コンテキストファイル | スキルディレクトリ                |
| ----------- | -------------------- | --------------------------------- |
| Claude Code | `CLAUDE.md`          | `.claude/skills/<skill>/SKILL.md` |
| Codex CLI   | `AGENTS.md`          | `.codex/skills/`                  |
| IBM Bob     | `AGENTS.md`          | `.bob/skills/<skill>/SKILL.md`    |

AGENTS.md の公式サイトでは、AGENTS.md をこう説明しています。

> Think of AGENTS.md as a README for agents
> A simple, open format for guiding coding agents

https://agents.md/

つまり、人間向けの README とは別に、AIエージェント向けの前提や作業ルールを書く場所、という位置づけです。ビルド方法、テスト方法、コードスタイル、PRのルールみたいなものはここに置くのがちょうどいいです。

AGENTS.md Starter Kit でも、主要なコーディングエージェントは AGENTS.md を読む方向に寄っている、と整理されています。

> Most major coding agents read `AGENTS.md` natively.

https://agentsmd.kurkalabs.dev/

なので、共通の指示ファイルを1つ選ぶなら、今のところ `AGENTS.md` を中心にするのが一番扱いやすいと考えています。

ここでいうスキルは、`SKILL.md` で定義するカスタム手順です。たとえば「PRレビューコメントを確認して直す」「リリースノートを書く」「社内ツールを決まった手順で実行する」みたいな、毎回プロンプトで説明したくない作業をまとめておけます。

便利なんですが、同じスキルを `.claude/skills/`、`.codex/skills/`、`.bob/skills/` にコピーし始めると、すぐに破綻します。

# AGENTS.md にまとめる

まず、プロジェクト共通の指示は `AGENTS.md` にまとめます。

Codex や IBM Bob は `AGENTS.md` をそのまま読めます。一方で Claude Code は `CLAUDE.md` を読むので、`CLAUDE.md` 側から `AGENTS.md` を import します。

Claude Code の Memory ドキュメントにも、そのまま書かれています。

> Claude Code reads `CLAUDE.md`, not `AGENTS.md`.

https://code.claude.com/docs/en/memory

ただし、同じページで `AGENTS.md` を使っているリポジトリでは `CLAUDE.md` から import する方法が紹介されています。Claude Code の import は `@path/to/import` 形式で、起動時に読み込まれる仕組みです。

```markdown
# CLAUDE.md

@AGENTS.md

## Claude Code 固有の設定

- コミット前に `npm test` を実行する
```

`@AGENTS.md` と書いておけば、Claude Code も同じ共通ルールを読めます。Claude Code 固有の追記があれば、その下に書けばOKです。

固有設定がまったく不要なら symlink でも動きます。

> A symlink also works

https://code.claude.com/docs/en/memory

```bash
ln -s AGENTS.md CLAUDE.md
```

ただ、あとから Claude Code だけの設定を書きたくなることがあるので、自分は `CLAUDE.md` に `@AGENTS.md` を書く方式にしています。

# スキルは .agents/skills に集約する

次にスキルです。

Agent Skills の説明では、スキルはこう定義されています。

> a skill is a folder containing a `SKILL.md` file

https://agentskills.io/

同じ説明では、スキルにはスクリプト、参考資料、テンプレートなども同梱できるとされています。つまり、`SKILL.md` だけを単体で管理するというより、ディレクトリ単位で管理するのが自然です。

Claude Code の Memory ドキュメントでも、常に読み込むルールと、必要なときだけ使うスキルは分けて考えられています。

> only load when you invoke them or when Claude determines they’re relevant

https://code.claude.com/docs/en/memory

この考え方に合わせると、プロジェクト全体の前提は `AGENTS.md`、繰り返し使う作業手順は `.agents/skills/`、という分け方がしっくりきます。

各ツールのスキルディレクトリをそのまま使うのではなく、実体を `.agents/skills/` に置きます。`.agents/` という名前にしているのは、特定のツール名に寄せたくなかったからです。Claude 用でも Codex 用でも Bob 用でもなく、エージェント共通の置き場、という扱いです。

```text
.agents/skills/          # 実体
.claude/skills  -> ../.agents/skills
.codex/skills   -> ../.agents/skills
.bob/skills     -> ../.agents/skills
```

セットアップはこんな感じです。

```bash
# 1. 共通のスキル置き場を作る
mkdir -p .agents/skills

# 2. 既存スキルを集約する
cp -r .bob/skills/* .agents/skills/
cp -r .claude/skills/my-skill .agents/skills/

# 3. 各ツールの skills を symlink に置き換える
rm -rf .bob/skills .claude/skills .codex/skills
ln -s ../.agents/skills .bob/skills
ln -s ../.agents/skills .claude/skills
ln -s ../.agents/skills .codex/skills
```

これで、新しいスキルは `.agents/skills/` に追加するだけになります。

```text
.agents/skills/create-release-note/SKILL.md
```

この1ファイルを置けば、対応しているツールから同じスキルを使えます。ツールを切り替えても「こっちには古い手順が残っていた」みたいなことが起きにくくなります。

# 重複スキルのマージだけ注意

すでに複数ツールでスキルを運用している場合、同名スキルがあるかもしれません。

その場合は、いきなり `cp -r` で上書きしないほうがいいです。まず差分を見ます。

```bash
diff -ru .claude/skills/review-pr .bob/skills/review-pr
```

どちらが新しいか、あるいは両方に必要な変更があるかを確認してから `.agents/skills/` に統合します。

ここだけは少し手作業が必要です。でも一度集約してしまえば、以後は `.agents/skills/` だけ見ればよくなります。

# Codex は設定で path を変える方法もある

Codex CLI は symlink ではなく、設定でスキルを指す方法もあります。Codex の config reference では `skills.config.<index>.path` はこう説明されています。

> Path to a skill folder containing `SKILL.md`.

https://developers.openai.com/codex/config-reference

設定上も「`SKILL.md` を含むスキルフォルダへのパス」を指定する形なので、`.agents/skills/<skill>` を指す運用と相性がいいです。

```toml
[[skills.config]]
path = ".agents/skills/create-release-note"
enabled = true
```

個別のスキルを明示的に有効化したい場合は、この方法も使えます。ただ、Claude Code や Bob も含めてディレクトリごと共有したい場合は、各ツールの標準ディレクトリを symlink にするほうが分かりやすいです。

自分は「ツールを追加するときは symlink を1本増やす」という運用にしています。

# やってよかったこと

この構成にしてから、AIコーディングエージェントを切り替える心理的コストがかなり下がりました。

以前は、新しいツールを試すたびに「このリポジトリのルールはどこに書いたっけ」「このスキルは Claude 側にしかなかったっけ」みたいな確認が必要でした。今は `AGENTS.md` と `.agents/skills/` を見れば済みます。

新しいツールを試したくなったら、基本はこれだけです。

```bash
mkdir -p .new-agent
ln -s ../.agents/skills .new-agent/skills
```

AGENTS.md とスキルディレクトリを持つエージェントなら、だいたい同じ考え方で増やせます。

# おわりに

AIコーディングエージェントは増えていくと思います。ツールごとに得意不得意があるので、しばらくは複数を使い分けることになりそうです。

そのときに、設定やスキルまでツールごとに分散していると、比較する前の準備で疲れます。`AGENTS.md` と `.agents/skills/` に集約しておくと、少なくとも「同じことを何回も書く」問題はかなり減ります。

個人利用なら symlink で十分です。チームで共有したくなったら、`.agents/skills/` を git submodule にしたり、プラグインとして配布したりする形に発展させるとよさそうです。
