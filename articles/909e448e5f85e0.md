# はじめに

本記事は、プラットフォームエンジニアリング (Platform Engineering) を調べ始めて、
本を読んでもピンと来ないので、手を動かして理解を深めようとした話です。
Platform Engineering / Internal Developer Portal (IDP) を詳しく知らないソフトウェアエンジニアが、
Backstage を「一度は触ってみようかな」と思える状態になることを目指します。
私はソフトウェアエンジニアとしての開発経験はある一方、
Platform Engineering / IDP は取り組み始めた段階です。

想定読者:

- Platform Engineering / IDP を初めて調べる人
- ソフトウェア開発の基礎は分かっている人

この記事で得られること:

- Platform Engineering / IDP / Backstage の位置づけ
- Backstage を試す入口（手を動かすための最短ルート）
- 触って気づいた、注意点

# Platform Engineering と Internal Developer Portal (IDP)

Platform Engineering は、
開発者の生産性・体験を向上するために、社内向けのプラットフォームを
「プロダクト」として設計・運用する考え方です。
CI/CD、テンプレート、ドキュメント、標準化された運用などを、
各チームが個別に整備するのではなく、共通の基盤として提供します。

https://www.gartner.co.jp/ja/articles/what-is-platform-engineering

また、社内開発者ポータル (IDP: Internal Developer Portal)は、
開発者がアプリケーション開発に必要な環境、ツール、プロセスなどを
統合的に提供する社内向けのプラットフォームのポータルサイトです。
「必要な情報や作業をここに集約する」というのがポイントです。

以下も詳しくてわかりやすいです。

https://speakerdeck.com/skitamura7446/the-future-of-platform-engineering-unveiling-the-true-value-of-containers-and-backstage

Backstage は IDP の OSS です。

https://backstage.io/

以下のようなことができます。

- ドキュメント整備 (TechDocs)
- マイクロサービス作成のテンプレ化 (Software Templates)
- サービス検索・所有者特定の効率化 (Catalog)

# なぜ Backstage を触ってみたのか

Platform Engineering のコア部分に Internal Developer Portal (IDP)がある。
IDP は「開発に必要な情報や作業の入口を一箇所に集約するポータル」で、
Backstage はその OSS 実装という位置づけ。
IDP の OSS として Backstage がある。一強。
Platform Engineering、本とか記事を読んで雰囲気わかったけど、
まだピンときてない気がしたので、手を動かして理解を深めようとした
ということで、Backstage を触ってみた。

# 最初に抱いていた誤解

Backstage だけ動かしたら IDP 完成。
違う。他のツールと連携しないとコンテンツが何もなくて何もわからない。
最低限、GitHub/GitLab と連携する必要ある。
IDP の価値は「情報や作業の集約」にあるので、
連携先がないと空っぽのポータルになってしまう。

# 試したこと

どこから、始めたらいいのかよくわからなかったので、調べたら、
以下の github が docker compose でサクッと試しやすくて良さそうで試してみた。

https://github.com/ap-communications/chocott-backstage

リンク先のドキュメントが丁寧なので、基本はその手順に従えば OK。

大まかな流れはこの程度で試せる。

- リポジトリを clone して docker compose で起動
- Backstage にアクセスして初期画面を確認
- GitHub と連携して Catalog に登録する

# 試してみた気づき

テンプレートの連携やドキュメントの管理を試せる。
IDP を構築するための、最初の足がかりになる。
docker compose で提供されているので DB とかはそんなに気にする必要ない。
最小構成で触るだけでも、IDP が「ハブ」として機能するイメージが掴めた。

進めていくとわかるけど、個人アカウントで連携するか、
Organization と連携するか選ぶ必要がある。
個人アカウントで連携したけど、Backstage の良さを体験するには、
最初から Organization でやればよかったなと思う。
Organization は自分でも作れるので、試しに作って連携するのが良さそう。
Platform Engineering がそもそも社内のプラットフォームをプロダクトとして扱う概念なので、チーム体制・チーム開発の話。なので、仕事でやるには、Organization としての連携が必須。Organization で連携した方がどんなことできるのかイメージしやすい。
Organization 連携は環境変数の設定が違うのと、ユーザー管理を試せるようになるはず。

GitHub を SSO 的使いつつ、「ハブ」としてテンプレートや
ドキュメント管理にするのは、まさに IDP でやろうとしていることで、
とても相性が良い。

GitHub が IDP の SaaS を出したらいいんじゃね？とも思った。

# おわりに

Backstage を試しに使ってみることで、どういうふうに IDP を利用・運用するのか、Platform Engineering をどのように組織に組み込んでいくべきか、理解が少し深まった気がします。

また、Portal が必要なのは変わらないと思うけれど、
AI Agent 時代に Platform Engineering が必要かどうかも、さらに考えていきたいと思っています。

以下のような記事もあって、ちゃんと考える必要がありそうです。
実際に、Backstage や Red Hat が出している Backstage のマネージドサービス Developer Hub では MCP としての機能も提供しています。
