# Platform Engineering と Internal Developer Portal と BackStage

Platform Engineering は、
**開発者の生産性・体験を向上させるために、社内向けのプラットフォームを「プロダクト」として設計・運用する考え方・手法**です。
CI/CD、テンプレート、ドキュメント、標準化された運用などを、
各チームが個別に整備するのではなく、共通の基盤として提供します。

https://www.gartner.co.jp/ja/articles/what-is-platform-engineering

また、社内開発者ポータル (IDP: Internal Developer Portal) は、
開発者がアプリケーション開発に必要な環境、ツール、プロセスなどを
統合的に提供する社内向けのプラットフォームの **ポータルサイト** です。
**「必要な情報や作業をここに集約する」** というのがポイントです。

以下も詳しくてわかりやすいです。

https://speakerdeck.com/skitamura7446/the-future-of-platform-engineering-unveiling-the-true-value-of-containers-and-backstage

最後に、Backstage は**IDP の代表的な OSS 実装**です。

https://backstage.io/

以下のようなことができます。

- ドキュメント整備 (TechDocs)
- マイクロサービス構築のテンプレ化 (Software Templates)
- サービス検索・所有者特定の効率化 (Catalog)

# なぜ Backstage を触ってみたのか

Platform Engineering では**IDP が重要**で、OSS では**Backstage が広く使われている**印象です。
本や記事で雰囲気はわかったものの、まだピンときていなかったので、
手を動かして理解を深めようとして Backstage を触ってみました。

# 最初に抱いていた誤解

**Backstage だけ動かしたら IDP が完成する** と誤解していました。
実際は、他のツールと連携しないとコンテンツが何もなくて全く雰囲気をつかめません。
最低限、**開発のコアとなっている GitHub/GitLab と連携**する必要があります。
GitHub/GitLab はコードやドキュメントやメンバーを管理しており、**開発者体験のコア**だと思います。
IDP の価値は **「情報や作業の集約」** にあるので、連携先がないと **空っぽのポータル**になってしまいます。

# 試したこと

最初、Backstage を clone しても期待通り動きませんでした。
どこから始めたらいいのかよくわからなかったので調べたところ、
以下の GitHub が docker compose でサクッと試しやすくて良さそうだったので、
試してみました。名前が良いです。

https://github.com/ap-communications/chocott-backstage

リンク先のドキュメントが丁寧なので、**基本はその手順に従えば大丈夫**です。

大まかな流れはこの程度で試せます。

- リポジトリを clone
- 環境変数を設定
- docker compose で起動
- Backstage にアクセスして操作（テンプレートを登録したり利用したり）

# 試してみた気づき

- [chocott-backstage](https://github.com/ap-communications/chocott-backstage) で試せること
  - テンプレートの連携やドキュメントの管理を試せます。
  - **IDP を構築するための最初の足がかり**になります。
  - docker compose で提供されているので DB などはあまり気にしなくてよいです。
  - 最小構成で触るだけでも、**IDP が「ハブ」として機能するイメージ**が掴めました。
- 連携先は GitHub Organization がおすすめ
  - GitHub 個人アカウントで連携するか、Organization と連携するかを選ぶ必要があります。
  - 個人アカウントで連携しましたが、Backstage の良さを体験するには最初から Organization でやればよかったなと思いました。
  - Organization は自分でも作れるので、試しに作って連携するのが良さそうです。
  - Platform Engineering は社内のプラットフォームをプロダクトとして扱う概念なので、**チーム体制・チーム開発の話**になります。
  - 仕事でやるなら**Organization 連携が前提**です。
  - Organization にすることで、**ユーザー管理**も試せそうです。
- テンプレートの構成が IDP の運用イメージに近い
  - [chocott-backstage](https://github.com/ap-communications/chocott-backstage) が準備しているテンプレートは、GitHub を SSO 的に使いつつ「ハブ」としてテンプレートやドキュメント管理しています。
  - これは IDP がやろうとしていることに近く、最初に試すにはちょうど良いように感じました。
- 余談: GitHub が IDP のマネージドサービスを出したらそれで良いかも、とも思いました。

# おわりに

Backstage を試しに使ってみることで、 **IDP の利用・運用のイメージ** や、
**Platform Engineering を組織に組み込む感覚** が少し掴めた気がします。

まずは最小構成で触って、IDP の実感を掴むのがおすすめです。
