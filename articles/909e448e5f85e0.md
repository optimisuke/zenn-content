# Platform Engineering と Internal Developer Portal (IDP)と BackStage

Platform Engineering は、
開発者の生産性・体験を向上させるために、社内向けのプラットフォームを「プロダクト」として設計・運用する考え方・手法です。
CI/CD、テンプレート、ドキュメント、標準化された運用などを、
各チームが個別に整備するのではなく、共通の基盤として提供します。

https://www.gartner.co.jp/ja/articles/what-is-platform-engineering

また、社内開発者ポータル (IDP: Internal Developer Portal) は、
開発者がアプリケーション開発に必要な環境、ツール、プロセスなどを
統合的に提供する社内向けのプラットフォームのポータルサイトです。
「必要な情報や作業をここに集約する」というのがポイントです。

以下も詳しくてわかりやすいです。

https://speakerdeck.com/skitamura7446/the-future-of-platform-engineering-unveiling-the-true-value-of-containers-and-backstage

最後に、Backstage は IDP の代表的な OSS 実装です。

https://backstage.io/

以下のようなことができます。

- ドキュメント整備 (TechDocs)
- マイクロサービス構築のテンプレ化 (Software Templates)
- サービス検索・所有者特定の効率化 (Catalog)

# なぜ Backstage を触ってみたのか

Platform Engineering では IDP が重要で、OSS では Backstage が広く使われている印象です。
本や記事で雰囲気はわかったものの、まだピンときていなかったので、
手を動かして理解を深めようとして Backstage を触ってみました。

# 最初に抱いていた誤解

Backstage だけ動かしたら IDP が完成すると誤解していました。
実際は、他のツールと連携しないとコンテンツが何もなくて何もわからないです。
最低限、開発のコアとなっている GitHub/GitLab と連携する必要があります。
GitHub/GitLab はコードやドキュメントやメンバーを管理しており、まさに開発者体験と IDP のコアになっています。
IDP の価値は「情報や作業の集約」にあるので、連携先がないと空っぽのポータルになってしまいます。

# 試したこと

始めるにしても、Backstage を clone しても期待通り動きませんでした。
どこから始めたらいいのかよくわからなかったので調べたところ、
以下の GitHub が docker compose でサクッと試しやすくて良さそうだったので、
試してみました。名前がいいです。

https://github.com/ap-communications/chocott-backstage

リンク先のドキュメントが丁寧なので、基本はその手順に従えば大丈夫です。

大まかな流れはこの程度で試せます。

- リポジトリを clone して環境変数を設定
- docker compose で起動
- Backstage にアクセスして初期画面を確認
- Backstage を操作（テンプレートを登録したり利用したり）

# 試してみた気づき

テンプレートの連携やドキュメントの管理を試せます。
IDP を構築するための最初の足がかりになります。
docker compose で提供されているので DB などはあまり気にしなくてよいです。
最小構成で触るだけでも、IDP が「ハブ」として機能するイメージが掴めました。

進めていくと、個人アカウントで連携するか、
Organization と連携するか選ぶ必要がある。
個人アカウントで連携したけど、Backstage の良さを体験するには、
最初から Organization でやればよかったなと思いました。
Organization は自分でも作れるので、試しに作って連携するのが良さそうです。
Platform Engineering は社内のプラットフォームをプロダクトとして扱う概念なので、
チーム体制・チーム開発の話になります。仕事でやるなら Organization 連携が前提です。
Organization で連携した方が何ができるのかイメージしやすいです。
Organization 連携は環境変数の設定が違い、ユーザー管理も試せそうです。

GitHub を SSO 的に使いつつ、「ハブ」としてテンプレートや
ドキュメント管理にするのは、まさに IDP でやろうとしていることで、
とても相性が良い。

GitHub が IDP の SaaS を出したら面白いかも、とも思った。

# おわりに

Backstage を試しに使ってみることで、IDP の利用・運用のイメージや、
Platform Engineering を組織に組み込む感覚が少し掴めた気がします。

また、ポータルが必要なのは変わらないと思うけれど、
AI Agent 時代に Platform Engineering が必要かどうかも、さらに考えていきたいと思っています。

以下のような記事もあって、ちゃんと考える必要がありそうです。
実際に、Backstage や Red Hat が出している Backstage のマネージドサービス Developer Hub では、
MCP としての機能も提供しているようです。

https://www.cnia.io/pek2025/blog/ai-native-platformengineering.md/
