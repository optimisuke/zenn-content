# はじめに

`gh skill` というコマンドがあるのは知っていたんですが、よくわかってないまま放置していました。ということで実際に試してみたら、だいたいわかってきたので整理しておきます。

自分の Claude Code スキルを公開して、別の環境からインストールできるかを確認するのが目的です。`gh skill` 自体まだプレビュー段階なので、変わる可能性はあります。

# gh skill とは

GitHub CLI の拡張コマンドで、エージェントスキル（Claude Code などが使う `SKILL.md`）を GitHub 経由で公開・インストールできます。公式ドキュメントはこちらです。

https://cli.github.com/manual/gh_skill

サブコマンドは5つあります。

```bash
gh skill publish   # スキルを公開（GitHub Release を作成）
gh skill install   # スキルをインストール
gh skill update    # インストール済みスキルを更新
gh skill search    # スキルを検索
gh skill preview   # インストール前に中身を確認
```

# スキルのリポジトリ構造

スキルは `skills/*/SKILL.md` パターンで自動検出されます。複数スキルをまとめて1リポジトリに置けます。

```
my-skills/
├── .gitignore
├── README.md
└── skills/
    ├── zenn-article-writer/
    │   └── SKILL.md
    └── hello-world/
        └── SKILL.md
```

`SKILL.md` の最小構成はこれだけです。

```yaml
---
name: skill-name # ディレクトリ名と一致させる
description: 何をするスキルか、いつ使うかを書く
license: MIT # 推奨フィールド
---
## 手順
...
```

`name` はディレクトリ名と一致していないとエラーになります。`description` はエージェントがスキル選択に使うので、「何をする」「いつ使う」を具体的に書くのが重要です。

# publish の流れ

```bash
# 1. リポジトリを作成
gh repo create my-skills --public

# 2. push
git add -A && git commit -m "add skills" && git push -u origin main

# 3. 検証だけ（エラー確認）
gh skill publish --dry-run

# 4. 公開（対話式でバージョンを聞かれる）
gh skill publish

# 非対話で公開したい場合
gh skill publish --tag v0.1.0
```

`gh skill publish` が裏でやっていること：

- `SKILL.md` の仕様チェック
- GitHub Release の作成（`gh release create` 相当）
- リポジトリに `agent-skills` トピックを付与

つまり GitHub Releases の薄いラッパーです。`git push` だけではインストールできないので注意。インストールは GitHub Release を参照するので、必ず publish が必要です。

## dry-run で事前確認

`--dry-run` を使うと、公開せずに検証だけできます。

```bash
$ gh skill publish --dry-run
warning  zenn-article-writer  recommended field missing: license
warning  no git remote found. Create a GitHub repository with: gh repo create
Dry run complete.
```

よく出る warning：

- `license` フィールドがない → frontmatter に `license: MIT` などを追加
- タグ保護ルールセットがない → GitHub の Settings > Rules > Rulesets で設定（後述）

## タグ保護ルールセット

`gh skill install --pin v0.1.0` のようにバージョンを固定してインストールした場合、タグが後から書き換えられると参照先が変わってしまいます。これを防ぐのがタグ保護ルールセットです。

GitHub API で追加できます。

```bash
gh api repos/<owner>/<repo>/rulesets \
  --method POST \
  --header "Content-Type: application/json" \
  --input - <<'EOF'
{
  "name": "Protect tags",
  "target": "tag",
  "enforcement": "active",
  "conditions": {
    "ref_name": {
      "include": ["refs/tags/v*"],
      "exclude": []
    }
  },
  "rules": [
    {"type": "deletion"},
    {"type": "non_fast_forward"}
  ]
}
EOF
```

# install とスコープ

```bash
# プロジェクトスコープ（デフォルト）→ .claude/skills/ に入る
gh skill install <owner>/<repo> <skill-name> --agent claude-code

# ユーザースコープ → ~/.agents/skills/ に入る
gh skill install <owner>/<repo> <skill-name> --agent claude-code --scope user
```

`--scope user` にすると `~/.agents/skills/` に入るので、油断すると既存スキルを上書きするかもしれません。同名スキルがある場合は事前に確認しておいた方が良さそうです。

# update と search

```bash
# インストール済みスキルを全更新
gh skill update --all

# スキルを検索
gh skill search zenn
```

`gh skill search` は publish 直後には出てきません。インデックス反映にしばらく時間がかかります（数時間単位のようです）。

新しいスキルを追加してリリースした場合、`update` では追加されず `install` が必要です。`update` は既存スキルの更新のみです。

セッションを再起動しないと新しくインストールしたスキルが認識されないのも地味につまずきました。

# おわりに

`gh skill publish` は GitHub Releases のラッパーで、仕組みがわかってしまえばシンプルです。スキルのリポジトリ構造も `skills/*/SKILL.md` パターンだけ覚えておけばとりあえず動きます。

グローバルとプロジェクトスコープの優先順位と、`~/.claude/skills` がシンボリックリンクという点は、自分のように既存スキルがある環境で試すとはまりやすいので気をつけてください。
