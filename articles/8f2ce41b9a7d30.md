# はじめに

LLM API を呼ぶアプリを作っていると、「あの生成、どのプロンプトで何が返ってきたんだっけ」を後から見返したくなります。自分の場合は、子ども向けワークショップ用の4コマ漫画アプリ（人が1・3コマ目を描き、AI が2・4コマ目を作る）を開発中で、プロンプトをあれこれ変えながら試しているうちに、**プロンプトを改善したいのに材料がない**、という状態になっていました。

:::message
アプリ自体の話は、そのうち別の記事に書くかもしれません。[このイベント](https://x.com/techtechtechls/status/2069975197105729911)で使う予定なので、気になる人はぜひ。
:::

で、LLM の可観測性ツールを調べ始めると、Langfuse、LangSmith、Helicone、Datadog LLM Observability…と乱立していて、**どれにのせるか決めきれない** んですよね。しかも自分は本番の送信先も決めていませんでした。最初は「Splunk の無料版でトレース見ればいいか」と思っていたら、無料版で見られるのは検索できるログまでで、トレースの可視化は有料の Observability Cloud 側の機能だと分かって、当初の想定が崩れました。

ということで、**ツール選定を保留したまま可観測性だけ先に確保する**、つまり「決めない」構成にしました。あとから Langfuse を足したら **アプリのコード変更が本当にゼロ** だったので、この構成は悪くなかった気がしています。整理しておきます。

:::message
**忙しい人向けまとめ**

- **やったこと**: LLM 呼び出しの記録（プロンプト・出力・トークン数・所要時間）を OpenTelemetry で取れるようにした。アプリに入れるのは **OTel の標準パッケージだけ**（Langfuse などのベンダー SDK はゼロ）で、計装コードは全部で230行くらい
- **結果**: あとから Langfuse を足すのが **アプリ変更ゼロ**（Collector の設定に送り先を1つ追加するだけ）で済んだ。本番の送信先（Splunk / Datadog / Instana など）も同じやり方で後から決められる
- **仕掛け①**: データの送り先はアプリに書かず、間に挟んだ **OTel Collector が振り分ける**。ローカルでは Grafana スタック（Tempo / Loki / Prometheus）に送って確認する
- **仕掛け②**: 記録に付ける属性名を、LLM 用の標準規約 **gen_ai semantic conventions**（`gen_ai.request.model` など）に合わせた。おかげで Langfuse 側がモデル名・トークン数・入出力を勝手に解釈してくれる
- **実装の方針**: HTTP まわりは自動計装に任せ、業務的な意味（どのコマの企画か、など）が要る LLM 呼び出しだけ `withAiSpan` という **ラッパー関数で包んで記録する**
:::

# やりたいこと

- LLM 呼び出し（企画・画像生成・画像解釈の4箇所）の **プロンプト全文・出力・トークン数・所要時間** を後から追える
- ローカルで完結して確認できる。本番は送信先未定のまま、**何も送らない**
- Langfuse のような LLM 特化ツールを **アプリを変えずに** あとから足せる
- 本番の送信先（Splunk・Datadog・Instana などを想定）はまだ決めない。決まったら、これも **アプリを変えずに** 置き換えまたは追加できる

最終的にこういう形になりました。

```text
アプリ（Next.js）
  │  OTLP で送るだけ（env 未設定なら何も送らない）
  ▼
OTel Collector  ← 送信先の知識はぜんぶここに集約
  ├── Tempo（トレース）─┐
  ├── Loki（ログ）      ├─ Grafana で見る
  ├── Prometheus（メトリクス）┘
  ├── Langfuse（LLM 分析）← あとから exporter 1つ追加
  └── 本番の送信先（Splunk / Datadog / Instana など・未定）← 決まったら置き換え or 追加
```

※ Tempo〜Langfuse まではローカル開発用です。本番の送信先はまだ決めていません。

ポイントは「**アプリは OTLP を喋るだけ、送信先の振り分けは Collector の責務**」という分担です。アプリに Langfuse SDK のようなベンダー固有のパッケージは1つも入れません。

# 前提: OpenTelemetry の登場人物

OpenTelemetry（OTel）は名前は知ってるけど自分で計装したことはない、くらいの人向けに、登場人物だけ先に整理しておきます。荷物の発送にたとえるとこうです。

| 部品 | たとえ | 役割 |
| --- | --- | --- |
| API（`@opentelemetry/api`） | 窓口 | 「ここからここまでを1つの作業として記録して」とお願いするインターフェースだけ。超軽量 |
| SDK（`@opentelemetry/sdk-node`） | 倉庫と作業員 | お願いを受けて実際に span を組み立て、溜めて、発送する本体 |
| Exporter | 発送係 | 溜まったデータをどこへ・どの形式で送るかを担当する部品 |
| Collector | 中継拠点 | アプリの外で受け取り、複数の宛先へ振り分けるプロセス（コンテナで動かす） |

ここで1つ注意なのが API と SDK の関係です。一般に「SDK」と聞くと、API を呼びやすく包んだラッパー（自分のコード → SDK → API の順）を想像すると思いますが、**OTel では逆** です。自分のコードが呼ぶのはあくまで API（インターフェース）で、SDK は **API の後ろに差し込まれる実装** です。起動時に SDK が「自分が実装だよ」と API に登録し、登録が無ければ API へのお願いは空振りして何も起きません。インターフェースと実装を分離する DI（依存性注入）に近い構造です。

この関係とデータの流れを図にするとこうなります。API・SDK・Exporter の3つは **アプリの Node.js プロセスの中** にいて、Collector だけが **アプリの外**（別プロセス、うちは Docker コンテナ）にいます。

```text
┌── アプリの Node.js プロセス ────────────────────────────┐
│                                                         │
│  自分のコード（route.ts など）                            │
│  「この範囲を1つの作業として記録して」とお願いするだけ       │
│      │                                                   │
│      ▼                                                   │
│  API（窓口）                                              │
│      │  ※ SDK が起動していなければ、お願いはここで空振り     │
│      │    して何も起きない（エラーにもならない）              │
│      ▼                                                   │
│  SDK（倉庫と作業員）… span を組み立てて、まとめて溜める      │
│      │                                                   │
│      ▼                                                   │
│  Exporter（発送係）… OTLP という共通形式に詰めて送り出す     │
│                                                          │
└──────┼───────────────────────────────────────────────────┘
       │ OTLP/HTTP（数秒ごとにバッチ送信）
       ▼
  Collector（中継拠点）← アプリが知っている宛先はここだけ
       ├──▶ Tempo（トレース）
       ├──▶ Loki（ログ）
       ├──▶ Prometheus（メトリクス）
       └──▶ Langfuse など … 宛先の追加・変更はここだけで完結
```

この図で伝えたいことは2つです。1つは、**自分のコードが直接触るのは一番上の「窓口」だけ**で、その先（SDK・Exporter）は起動時に一度設定したら意識しなくていいこと。もう1つは、**アプリが知っている送信先は Collector 1つだけ**で、その先に何があるか（Tempo なのか Langfuse なのか Splunk なのか）をアプリは一切知らないことです。この2つ目が、この記事のテーマである「決めない」を支えています。

もう1つ大事なのが **semantic conventions（属性の命名規約）** です。LLM 呼び出し用には `gen_ai.*` という規約があり、`gen_ai.request.model` や `gen_ai.usage.input_tokens` のような属性名が決まっています。この規約に寄せておくと、あとで出てくる Langfuse のように **規約を理解するツールが属性を勝手に解釈してくれる** ようになります。

# アプリ側の計装は3層・全部で230行くらい

アプリ（Next.js App Router + openai パッケージで Azure OpenAI を呼ぶ構成）に入れた可観測性コードは、3層に分かれています。

## 層1: 起動基盤（46行）— 入れるだけで効く層

Next.js には `instrumentation.ts` という起動時フックがあるので、そこで SDK を起動します。

```ts:instrumentation.ts
export async function register() {
  if (process.env.NEXT_RUNTIME !== "nodejs") {
    return;
  }
  if (!process.env.OTEL_EXPORTER_OTLP_ENDPOINT) {
    return;
  }
  const { startOtel } = await import("./lib/otel");
  startOtel();
}
```

env が未設定なら SDK を import すらしないので、**本番（送信先未定）は env を置かないだけで完全に無効** になります。

SDK の起動側はこんな感じです。

```ts:lib/otel.ts（抜粋）
import { NodeSDK } from "@opentelemetry/sdk-node";
import { OTLPTraceExporter } from "@opentelemetry/exporter-trace-otlp-http";
import { UndiciInstrumentation } from "@opentelemetry/instrumentation-undici";

export function startOtel() {
  const sdk = new NodeSDK({
    // fetch(undici) の自動計装。LLM への実 HTTP リクエストが子 span で見える
    instrumentations: [new UndiciInstrumentation()],
    traceExporter: new OTLPTraceExporter(), // OTEL_EXPORTER_OTLP_ENDPOINT を読む
    // ログとメトリクスの exporter も同様に設定（記事では省略）
  });
  sdk.start();
}
```

1点だけ Next.js 固有の注意があって、OTel SDK はバンドルされると計装が壊れるため、`next.config.ts` の `serverExternalPackages` に `@opentelemetry/sdk-node` を足しておく必要があります（Next.js 公式ドキュメントにも書かれています）。

ここまでで、Next.js 組み込みの route span（`POST /api/plan` など）と、外向き HTTP の span が自動で出ます。**アプリのロジックに1行も触れずに、「どの API が遅いか」まではこの層だけで見えます**。

## 層2: ヘルパー（126行）— gen_ai 規約を閉じ込める層

ただ、自動計装だけだと「これは4コマ目の企画で、プロンプトはこれで、トークンをいくら使った」という **業務的な意味** が取れません。そこで、LLM 呼び出しを包むヘルパーを1ファイル作りました。

```ts:lib/telemetry.ts（骨子）
import { metrics, Span, SpanStatusCode, trace } from "@opentelemetry/api";
import { logs, SeverityNumber } from "@opentelemetry/api-logs";

const tracer = trace.getTracer("my-app");

// AI 呼び出し1回分のメタ情報。呼び出し側はこれを組み立てて渡すだけ
export type AiCallInfo = {
  kind: string;             // plan_panel2 / image_panel4 / interpret など
  model: string;
  draftId?: string | null;  // 作品ID（4コマ横断で追うためのキー）
  position?: number;        // 何コマ目か
  prompt?: string;          // 合成済みプロンプト全文
};

export async function withAiSpan<T>(info: AiCallInfo, fn: (span: Span) => Promise<T>): Promise<T> {
  return tracer.startActiveSpan(
    `ai ${info.kind}`,
    {
      attributes: {
        "gen_ai.operation.name": info.kind,       // plan_panel2 など
        "gen_ai.request.model": info.model,
        "gen_ai.prompt": truncate(info.prompt),   // 4000字で切り詰め
        "app.draft_id": info.draftId ?? "",
      },
    },
    async (span) => {
      try {
        const result = await fn(span);
        span.setStatus({ code: SpanStatusCode.OK });
        return result;
      } catch (error) {
        span.recordException(error as Error);
        span.setStatus({ code: SpanStatusCode.ERROR });
        throw error;
      } finally {
        span.end();
      }
    },
  );
}

// 出力テキストとトークン数を span・ログ・メトリクスに記録する
export function recordAiCompletion(span: Span, info: AiCallInfo, output: { text?: string; usage?: Usage }) {
  span.setAttribute("gen_ai.usage.input_tokens", output.usage?.prompt_tokens ?? 0);
  span.setAttribute("gen_ai.usage.output_tokens", output.usage?.completion_tokens ?? 0);
  span.setAttribute("gen_ai.completion", truncate(output.text ?? ""));
  // 同じ内容をログ（Loki 行き）とメトリクス（トークン数ヒストグラム）にも記録
}
```

属性名を `gen_ai.*` 規約に寄せているのがミソで、**規約の知識はこのファイルに全部閉じ込めています**。呼び出し側は命名規約を知りません。

ここで大事なのが、このファイルの import は `@opentelemetry/api` と `@opentelemetry/api-logs` の **API パッケージ2つだけ** という点です。前提のところで書いたとおり、SDK が起動していなければ API へのお願いは空振りする（いわゆる no-op）ので、**SDK 未起動の本番でもこのコードはそのまま安全に動き、「観測が有効か？」という分岐がアプリのどこにも要りません**。これはありがたい設計です。

ちなみにこの空振りは自分で実装するものではなく、API パッケージに最初から入っています。SDK 未起動のときに返ってくる span の実体は `NonRecordingSpan` というクラスで、`node_modules` の中を覗くとこうなっています。

```js
// @opentelemetry/api の NonRecordingSpan（抜粋）
class NonRecordingSpan {
  // By default does nothing
  setAttribute(_key, _value) {
    return this;
  }
  // By default does nothing
  addEvent(_name, _attributes) {
    return this;
  }
  end(_endTime) {}
}
```

メソッドは本物と同じ顔をしていて中身だけが空、という作りです（Null Object パターン）。呼ぶ側からは本物と区別がつかないので、**ラップした処理（LLM 呼び出し本体）は普通に実行されて結果も普通に返り、記録だけがどこにも残らない**、という動きになります。

最後に `AiCallInfo` の中身について。`kind`・`model`・`prompt` はどの LLM アプリでもだいたい共通ですが、`draftId`・`position` は「4コマ漫画の作品を横断で追いたい」といううちのアプリの都合です。**別のアプリで真似する場合は、ここを自分のドメインのキーに差し替えれば OK** です（EC なら注文 ID、チャットなら会話 ID など）。属性名は `gen_ai.*` とぶつからないように `app.*` みたいな自分の名前空間に置くのがおすすめです。

## 層3: LLM 呼び出し4箇所を各15行ほどでラップ

最後に、実際に LLM を呼ぶ Route Handler（`app/api/plan/route.ts` など4ファイル）で、呼び出しをヘルパーで包みます。

```ts:app/api/plan/route.ts（抜粋）
const spanInfo = {
  kind: "plan_panel2",          // 2コマ目の企画
  model: "gpt-5-4-mini",
  draftId,
  prompt: prompt.content,       // 合成済みプロンプト全文
};

const output = await withAiSpan(spanInfo, async (span) => {
  const completion = await azureOpenAI().chat.completions.create({ /* 省略 */ });

```

1箇所あたり15行くらいの diff で、これを4箇所。合計すると、起動基盤46行 + ヘルパー126行 + 呼び出し4箇所×15行 = **230行前後がこのアプリの可観測性コードの全量** です。package.json に増えたのは `@opentelemetry/*` の11パッケージだけで、**ベンダー固有のパッケージはゼロ** です。

# なぜ自動計装ライブラリを使わなかったか

層3を見て「openai パッケージの自動計装ライブラリを使えば、この手書きは要らないのでは？」と思った人がいると思います。実際、選択肢はいくつかあります。

| アプローチ | アプリの変更 | 業務的意味の付与 | 気になる点 |
| --- | --- | --- | --- |
| 自動計装ライブラリ（OpenLLMetry、instrumentation-openai） | ほぼゼロ | 結局手で足す | openai パッケージの内部に追従し続ける必要。画像 API のカバーが弱い |
| LLM クライアントを1箇所だけラップ | 少ない | 引数で渡す必要 | 呼び出し側の文脈（どのコマの企画か）を結局渡すので、コードの場所が変わるだけ |
| LLM ゲートウェイ（LiteLLM Proxy など） | ゼロ | ヘッダー等で工夫 | インフラが1ホップ増える。アプリ内 span と切れる |
| OTel 対応フレームワークに乗り換え（Vercel AI SDK など） | 大きい | 組み込みあり | 観測のためにクライアントを替えるのは本末転倒感 |
| アノテーション/デコレータ（Java の `@WithSpan` 相当） | 少ない | 結局中で足す | TS のデコレータはクラス必須で、Route Handler の平関数と噛み合わない |

アノテーション方式について少し補足すると、OTel 公式にも Java なら `@WithSpan`、Python ならデコレータの口があって、それらの言語では自然な選択肢です。ただ span に付けたい情報（draftId、コマ番号、合成後のプロンプト、呼び出し後に確定するトークン数）は **実行時にしか分からない値** なので、静的に貼るアノテーションだけでは足りず、結局「現在の span に属性を足す」コードを中に書くことになります。それなら最初から関数で包んで span を受け取る方がデータの流れが素直です。`withAiSpan` は、デコレータパターンを TypeScript の平関数スタイルで表現したもの、と捉えています。

どれを選んでも「これは4コマ目の企画」という **業務的な意味の付与からは逃げられない** ので、それなら薄いヘルパー + 明示的なラップが、依存最小でカバレッジ最大（画像生成 API も、ローカル用のモックモードも同じ形で計装できる）だと判断しました。

もう1つの判断軸は **依存の安定度** です。OTel の中でも安定度に濃淡があって、

- `@opentelemetry/api` は 1.x で何年も安定している
- `@opentelemetry/sdk-node` はまだ 0.x で、実は今回の実装中にもコンストラクタの引数形式が変わっていて型エラーを踏みました
- `gen_ai.*` の semantic conventions 自体もまだ発展途上

という状況です。なので「**一番安定した層（API）だけに広く依存し、変化の速い層（SDK、gen_ai 規約）への依存は1ファイルに隔離する**」という置き方にしました。実際、SDK の変更を踏んだときの修正は隔離した `lib/otel.ts` の1行で済んで、LLM を呼ぶ側のコードは無傷でした。

ただ、この選択は **将来の乗り換えを妨げません**。自動計装ライブラリ側も gen_ai 規約に寄ってきているので、安定したなと思ったら `withAiSpan` の中身を差し替えるだけで移行できるはずです。

# ローカルで見る: Grafana スタック

ローカルの確認用に、docker-compose にプロファイルを1つ足して、OTel Collector + Tempo（トレース）+ Loki（ログ）+ Prometheus（メトリクス）+ Grafana を立てました。構築自体は定番構成なので詳細は省きますが、要点だけ。

- Collector が 4318 番で OTLP を受け、トレース→Tempo、ログ→Loki、メトリクス→Prometheus に振り分ける
- Grafana のデータソースと「トレース→同じ trace_id のログへジャンプ」の相関は provisioning で自動設定
- アプリ側は `.env.local` に `OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318` を書くだけ

これで何が見えるかというと、たとえば2コマ目の画像生成のトレースはこうなります。

![Grafana Tempo のトレース画面。POST /api/generate の下の ai image_panel2 span を開くと、gen_ai の属性にプロンプト全文が見える](/images/8f2ce41b9a7d30/tempo-trace.png)
*画像生成リクエスト17.2秒のうち17.1秒が LLM の呼び出し。span の属性を開くと、どのコマ（app.panel_position）に、どんなプロンプトを投げたかまで見える*

span の属性にプロンプト全文・出力・トークン数が付いているので、**「この生成、なんでこうなった？」を span 1つで追えます**。ログ側（Loki）にも同じ内容が trace_id 付きで流れているので、draftId（作品ID）でフィルタして「この作品の4コマで何を投げて何が返ったか」を時系列で見る、みたいな使い方もできます。

# Langfuse をあとから足したら、アプリの変更がゼロだった

ここからが本題というか、この構成にしてよかったと思えた瞬間の話です。

実は最初に作ったのは前のセクションの Grafana スタックまでで、Langfuse は入っていませんでした。計装 → Grafana スタックで確認、までを作って一区切りにしていたんですが、使っているうちに、やっぱり LLM 専用の分析画面（プロンプトの比較、モデル別のトークン集計、Generation の一覧）が欲しくなってきます。そこで後日、Langfuse のセルフホスト版を docker-compose に足しました。

このとき **アプリ側の変更は本当にゼロ** で、やったことは Collector の設定に exporter を1つ足しただけです。

```yaml:otel-collector.yaml（追加分）
exporters:
  otlp_http/langfuse:
    endpoint: http://langfuse-web:3000/api/public/otel
    headers:
      Authorization: "Basic <base64(publicKey:secretKey)>"

service:
  pipelines:
    traces:
      exporters: [otlp_grpc/tempo, otlp_http/langfuse] # Langfuse を追加
```

これでなぜ成立するかというと、Langfuse が OTLP の受け口を持っていて、**gen_ai 規約の属性をそのまま解釈してくれる** からです。層2で規約に寄せておいた `ai plan_panel2` span が、Langfuse 側では Generation（モデル・トークン数・入出力付き）として認識されました。

![Langfuse の Generation 画面。model、トークン数、コスト、input にプロンプト、output に生成結果が構造化されて表示される](/images/8f2ce41b9a7d30/langfuse-generation.png)
*Langfuse 専用の計装は書いていないのに、Generation として認識される。モデル・トークン数（932 → 25）に加えて、コスト（$0.000811）まで自動で出た*

同じトレースが Tempo にも並行して届くので、使い分けはこうなりました。

- **Grafana**: リクエスト全体のウォーターフォール、非 LLM のログ・メトリクス
- **Langfuse**: LLM 呼び出しの分析、プロンプト改善の検討

なお Langfuse のセルフホストは正直重いです（web/worker に加えて ClickHouse、Redis、MinIO、Postgres でコンテナ6個）。手軽に試すなら Langfuse Cloud の無料枠に同じ形で送る手もあります。その場合も変わるのは Collector の endpoint とキーだけです。

# ハマりどころ

- **Langfuse の OTLP 受け口はトレースのみ**。ログとメトリクスは従来どおり Loki / Prometheus 行きです
- **メトリクスの unit はメトリクス名に付かないことがある**。`{token}` のような波かっこ付き unit は Prometheus のメトリクス名に反映されず、`gen_ai_client_token_usage_sum` という名前になります。最初 `_token_sum` を探して「届いてない！」と焦りました
- **Collector の設定変更は restart が必要**。bind mount でファイルを差し替えただけでは反映されません
- **プロンプトのテレメトリへの記録は切り詰めた方がいい**。うちは4000字で truncate しています（全文は元々 DB に保存しているので、テレメトリ側は改善検討に足る範囲だけ）
- **SDK が 0.x であることは受け入れる**。破壊的変更は起きます。起きたときの影響範囲を1ファイルに閉じ込めておくのが現実的な防御だと思います

# おわりに

「どのツールにのせるか」を決めないまま、**プロンプト改善の材料（プロンプト・出力・トークン・所要時間）が取れるようになりました**。決めなかったおかげで、**Langfuse は「試しに足す」ができたし、外すのも Collector の設定1行** です。今の Grafana スタック + Langfuse はあくまでローカル開発用で、本番の送信先は相変わらず決めていません。ただ、Splunk（Observability Cloud）も Datadog も Instana も OTLP の受け口を持っているので、**どれに転んでも env を1つ設定して Collector の exporter を置き換えるか足すだけ** のはずです（ここはまだ未検証なので、やったら別の記事に書くかもしれません）。

LLM まわりのツールは当分乱立が続きそうなので、「**アプリは OTLP と gen_ai 規約だけに寄せて、ベンダーの選択はアプリの外に追い出す**」という置き方は、長期的に見ると安定しそうな気がしています。

もっといいやり方があれば教えてください。
