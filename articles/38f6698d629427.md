最近見つけた Mozilla AI の[wasm-agents-blueprint](https://github.com/mozilla-ai/wasm-agents-blueprint)がなかなか面白くて触ってみた。

ブラウザ上で直接 Python AI エージェントを実行してて、興味深い。

動かすとこんな感じ。

![](/images/2025-07-07-09-28-39.png)

## どういうこと？

今までの AI アプリって、だいたいこんな感じやった：

```
フロントエンド → バックエンドサーバー → OpenAI API → レスポンス
```

でも、これは：

```
ブラウザ（Pyodide + Python） → OpenAI API → レスポンス
```

サーバーがない。これがどういうことか説明してみる。

## Pyodide って何？

[Pyodide](https://pyodide.org/)は、Python を WebAssembly（WASM）でブラウザ上で動かすやつ。要するに：

- ブラウザで Python コードが実行できる
- NumPy、Pandas、scikit-learn とかの有名どころも使えるらしい
- クライアント側のコード（HTTP とか JavaScript）の中で**micropip**ってのを使って PyPI からパッケージをインストールできる

で、今回知ったのが、**OpenAI Agents ライブラリまでブラウザで動く**ってこと。

## 実際のコードはこんな感じ

初期化部分：

```javascript
// Pyodideを読み込み
pyodide = await loadPyodide();

// micropipでopenai-agentsをインストール
await pyodide.runPythonAsync(`
import micropip
await micropip.install("openai-agents")
`);
```

エージェント実行部分：

```python
from agents import Agent, Runner, ModelSettings
import os

# APIキーを設定
os.environ['OPENAI_API_KEY'] = "your-api-key"

# エージェントを作成
agent = Agent(
    name="Assistant",
    instructions="あなたは親切なアシスタントです",
    model_settings=ModelSettings(extra_args={"timeout": 30})
)

# 実行
result = await Runner.run(agent, "こんにちは！")
```

これがブラウザで動く。JavaScript 関数の中で Python コードを実行して、OpenAI API を直接呼び出してる。

## ちょっと気になったこと

インストールは `openai-agents` なのに、インポートは `agents` になってる：

```python
# インストール時
await micropip.install("openai-agents")

# インポート時
from agents import Agent, Runner, ModelSettings
```

これは Python パッケージの命名規則で、PyPI パッケージ名とインポート名が違うのはよくある話らしい（例：`pip install scikit-learn` → `import sklearn`）。

## 何がいいん？

### サーバーレスがかなりサーバーレス

今までの「サーバーレス」って、サーバー管理をクラウドにお任せするってことやった。でもこれは本当にサーバーがない。

- バックエンドサーバー不要
- 静的サイトとしてデプロイできる
- CDN で配信するだけで OK

### プライバシーが良い感じ

ユーザーのプロンプトがサーバーを経由しない。ブラウザから直接 OpenAI API を呼び出すから、良くも悪くもサービス提供者のサーバーにデータが残らない。

### レスポンスが早い

サーバーを経由しないから、ネットワークの往復が減る。

### 開発しやすい

Python の AI ライブラリがそのままブラウザで使える。JavaScript で頑張らんでも済む。

## でも課題もある

### パフォーマンス面

- **初回読み込み時間**: Pyodide とライブラリのダウンロードで数十 MB（ライブラリによるけど）
- **メモリ使用量**: ブラウザのメモリをけっこう食う
- **実行速度**: WebAssembly やけどネイティブ Python より遅いとは思う

### 開発・運用面

- **API キー管理**: クライアントサイドで API キー扱うのはかなり制約が大きい
- **エラーハンドリング**: ネットワークエラーの処理が面倒かも
- **デバッグ**: ブラウザで Python デバッグするのは慣れが必要そう

### 使いやすさ

一般ユーザーに API キーの設定や Ollama の導入をお願いするのは、なかなかハードルが高い。

## 今どんなところで使えそう？

### 使えそうなケース

**開発者・研究者向け**

- AI エージェントの試作
- 教育・学習用途
- 技術デモ

**社内利用**

- 社内の AI 実験環境
- 研究開発チーム向けツール

### しんどそうなケース

**一般ユーザー向けサービス**

- IT リテラシーが高くないと使えない
- API キー管理が個人に委ねられる

**大規模・商用サービス**

- セキュリティ要件

## これからどうなりそう？

この技術が本領発揮するのは、ブラウザがネイティブで AI をサポートするようになったときかも。

### ブラウザネイティブ AI

Chrome、Edge、Safari とかが、ローカル LLM の API を提供するようになったら：

- API キー不要
- オフライン動作
- プライバシー完璧
- 高速レスポンス

### WebGPU との組み合わせ

WebGPU を使った高速推論ができるようになれば、ブラウザ上での AI 処理がもっと実用的になりそう。

## まとめ

今の wasm-agents-blueprint は、技術デモとしては面白いけど、実用的なサービスにするにはまだ課題が多い。

作者も「素晴らしいアイデアか面白いハックかまだ分からない」と言ってる。

でも、将来的には、ブラウザの AI 機能が充実してきたら、クライアントサイドでの AI 処理が普通になる可能性もある。

そういう意味で、今のうちから触っておくと面白いかも。
