# はじめに

ChatGPT とやりとりしていると、基本的にはキーボード入力やスマホのフリック入力でテキストを打つことになります。

それはそれで便利なんですが、最近、自分の中で音声入力を使う場面が少しずつ増えてきました。ちょっとしたメモ、長めの相談、頭の中にあることをそのまま吐き出したいとき。キーボードで打つより、声で話したほうが速いことがあります。

キーボードで入力していても同じなんですが、テキストだけだとそのときの気分や温度感はあまり残りません。せっかく音声を使うなら、そのへんの情報も少し拾えるとよさそうです。

たとえば、

```text
このエラーの原因を教えてください
```

というテキストだけでも意味は伝わります。でも、実際に声で話しているときには、焦り、不安、困惑、あきらめ、期待みたいなものも少し乗っています。

音声はテキストより情報量が多いはずです。とはいえ、その音声を毎回そのままマルチモーダル LLM に渡すほどではない気もしています。コンピューティングリソース的にも、速度的にも、普段のチャット入力としては少し重い。

そこで、音声そのものではなく、**感情を文字や絵文字にして、テキストと一緒に渡す**くらいがちょうどいいんじゃないかと思いました。

今回は AmiVoice API を使って、ChatGPT の入力欄に「感情付き音声入力」をする Chrome 拡張を試作しました。

最終的には、音声で話すとこういうテキストが入力欄に入ります。

![](/images/a3f7c2d9e8b1a4/final-emotion-input.png)

```text
😰 不安・困惑

このエラーの原因を教えてください
```

# 作ったものと狙い

作ったものはシンプルです。

ChatGPT のページ上に録音パネルを出して、話した内容を AmiVoice API で文字起こしし、感情分析の結果から代表感情を選んで、入力欄に絵文字付きで挿入します。

LLM はテキストだけでもかなり文脈を読んでくれます。ただ、同じ文章でも、話している人の状態によって返してほしい反応は少し違うと思っています。

`このエラーの原因を教えてください` という入力でも、落ち着いて聞いているのか、かなり焦っているのかで、うれしい返答は変わります。人間同士なら声の調子からなんとなく拾っている情報ですが、テキストだけにすると落ちます。

なので今回は、音声は AmiVoice API に渡して、ChatGPT にはテキストと感情ラベルだけ渡す形にしました。

コードはこちらに置いています。

https://github.com/optimisuke/hello-amivoice

# AmiVoice API を使った理由

今回のコンテストテーマが「音声認識 AmiVoice API と生成AIで作る音声体験」だったので、AmiVoice API を触ってみました。

https://zenn.dev/contests/zennfes-spring-2026-amivoice?tab=overview

文字起こしだけなら macOS 標準の音声入力もありますし、Whisper などの選択肢もあります。なので今回は、最初から「音声から感情っぽい情報も取れるか」を見ていました。

そこで AmiVoice API のドキュメントを見ていたら、感情分析も取れることが分かりました。

https://docs.amivoice.com/amivoice-api/manual/sentiment-analysis/

これは今回やりたいことにかなり合っています。

音声をテキストにするだけではなく、その音声から「不安っぽい」「怒っているっぽい」「期待しているっぽい」みたいなコンテキストを取れる。これを LLM への入力に混ぜると、少し会話が自然になる気がしました。

# 実装の全体像

最終的には、こんな構成にしました。

```text
ChatGPT ページ
  └─ content script
      ├─ ページ右下に録音オーバーレイを表示
      ├─ WebSocket でリアルタイム文字起こし
      └─ Stop 後に非同期HTTPで感情分析

AmiVoice API
  ├─ WebSocket: リアルタイム文字起こし
  └─ 非同期HTTP v1: 感情分析
```

最初は別タブに Recorder 画面を開く形にしていました。ただ、使い勝手としては ChatGPT の画面上で完結してほしかったので、最終的には content script で小さなオーバーレイを出す形にしました。

![](/images/a3f7c2d9e8b1a4/overlay-recording.png)

# 実装で大事だったこと

作りながら一番大事だと思ったのは、**文字起こしと感情分析を同じ速度で考えない**ことでした。

AmiVoice API のリアルタイム文字起こしは WebSocket でできます。

一方で、感情分析はドキュメント上、非同期 HTTP インタフェースで使う機能です。しかも、現時点では v2 ではなく v1 を使う必要があります。

https://docs.amivoice.com/amivoice-api/manual/async-http-interface/

なので、UI はこういう流れにしました。

録音中はリアルタイムに文字起こしを出し、Stop 直後はまず文字起こしだけを入力欄に入れます。その後、感情分析が返ってきたら絵文字付きテキストに置き換えます。

実際の挙動としては、Stop した瞬間にこう入れます。

![](/images/a3f7c2d9e8b1a4/pending-emotion.png)

```text
... 感情分析中

台風来てるから嫌なの
```

その後、感情分析が返ってきたら置き換えます。

```text
😰 不安・困惑

台風来てるから嫌なの
```

最初は感情分析が返ってくるまで待ってから挿入していました。ただ、非同期 HTTP は `queued` のまま待つことがあります。入力体験としては、ここで待ち時間を見せすぎないほうがよさそうだったので、「すぐ使えるテキスト」と「後から付く感情」を分けました。

# ハマったところ

一番ハマったのは音声フォーマットです。最初は `MediaRecorder` で録音して、そのまま `webm/opus` を AmiVoice API に投げていました。でも、うまく認識されず、

```text
recognition result is rejected because confidence is below the threshold
```

のようなエラーになりました。

手元のサンプル WAV を `curl` で送ると通るのに、拡張で録音した音声だと通らない。ここでしばらく悩みました（codexくんが）。結局、ブラウザ側で音声を `16kHz mono PCM WAV` にして送る形にしました。

https://docs.amivoice.com/amivoice-api/manual/getting-started/

録音は Web Audio API で取り、リアルタイム認識用には `16kHz mono PCM` のチャンクを WebSocket に流します。Stop 後には同じ音声を WAV にして、`sentimentAnalysis=True` を付けて非同期 HTTP に送ります。

Chrome 拡張としては、Vite の content script でもハマりました（codexくんが）。普通にビルドすると先頭に `import` が残ってしまい、

```text
Cannot use import statement outside a module
```

になりました。

Chrome の `content_scripts` は通常の script として読み込まれるので、content script だけ IIFE 形式で別ビルドするようにしました。

あと、最初は `ScriptProcessorNode` を使っていたんですが、Chrome から非推奨の警告が出ました。

```text
The ScriptProcessorNode is deprecated. Use AudioWorkletNode instead.
```

なので `AudioWorkletNode` を使うようにしました（codexくんが）。ただし、content script から worklet ファイルを読むには、`web_accessible_resources` も設定する必要がありました。

# 感情ラベルはまだ雑

AmiVoice API から返ってくる感情分析結果を、そのまま人間向けラベルにするのは少し考える必要があります。今回は MVP なので、返ってきた各スコアを雑に集約して、代表感情を1つ選ぶようにしました。

```text
😀 楽しい
😰 不安・困惑
😞 不満
😡 怒り
🤩 期待
```

正直、ここはまだ改善余地が大きいです。「あいうえお」みたいな感情のない発話でも、それっぽいラベルが付いてしまうことがあります。感情分析のスコアが弱いときはラベルを付けない、もしくは「ニュートラル」にするほうが自然かもしれません。

また、LLM に渡すなら、絵文字だけでなく、もう少し説明的にしてもよさそうです。

```text
感情: 不安・困惑
発話: このエラーの原因を教えてください
```

今は ChatGPT の入力欄で自然に見えるように、絵文字とラベルを先頭に置いています。

# おわりに

AI との会話は、今のところテキストが中心です。

でも、人間が話すときには、テキスト以外の情報もけっこう含まれています。全部を扱うのは重いけど、少しだけ感情として渡せると、AI とのやりとりがちょっと自然になる気がしました。

今回やりたかったのは、音声をそのまま生成AIに渡すことではなく、生成AIに渡す前に「音声を少し構造化する」ことでした。文字起こしと感情分析を同じ API で試せたので、その層をかなり作りやすかったです。

まだ UI/UX は改善が必要です。特に、感情分析が返る前にユーザーが入力欄を編集した場合の扱いはもう少し考えたいです。

それでも、普段の AI チャット入力に「声の温度感」を少し足す方向性としては、いい感じな気がしています。
